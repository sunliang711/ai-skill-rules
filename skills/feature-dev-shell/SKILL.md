# Shell 功能开发 SOP

> 当用户提出“写个 Shell 脚本”、“补一个部署脚本/自动化脚本”等请求时使用。

## 前置：需求澄清

> 必须先执行 `requirement-clarify-shell`，至少确认 `FS-01` ~ `FS-04`。
> 若脚本会产生写入副作用，必须在脚本内部添加写前前置检查逻辑。

## 阶段一：理解需求与方案设计

### 步骤 1：整理已知信息
- 脚本目的、运行环境、调用方式
- 输入、输出、依赖命令、环境变量与副作用
- 危险操作、权限要求、路径约束

### 步骤 2：先输出实现方案

```markdown
## 实现方案
- 脚本类型：bash / sh / zsh / CI helper / deploy script
- 入口参数：
- 依赖命令：
- 关键流程：
- 副作用与安全保护：
- 验证方式：
```

### 步骤 3：等待确认
- 用户确认后再实现
- 若涉及破坏性操作，必须再次明确保护措施

## 阶段二：脚本实现

> 编码时必须遵循 `rules/13-shell-scripting.mdc`；涉及命令执行、删除、下载、权限、密钥和路径安全时，还必须遵循 `rules/14-shell-security.mdc`。

### 步骤 4：按推荐顺序实现
- 明确 shebang 与严格模式
- 先补参数解析、帮助信息、依赖命令检查
- 再实现函数拆分、日志输出和核心命令链
- 临时文件和目录优先使用 `mktemp` + `trap`
- 危险操作优先提供 guard 条件、确认提示或 `--dry-run`
- 禁止把用户输入直接拼接进 `eval`、`bash -c`、`sh -c`

## 阶段三：验证与交付

### 步骤 5：验证要求
- 优先运行 `shellcheck`
- 复杂脚本优先补 `bats` 或项目既有测试
- 至少验证正常路径、错误路径和危险操作保护
- 明确脚本是否需要 dry-run 演练

### 步骤 6：输出交付摘要

```markdown
## 变更摘要
- 新增 / 修改脚本：
- 入口参数：
- 依赖命令：
- 保护措施：
- 验证情况：
```

### 步骤 7：任务交付文档
在 `docs/delivery/` 下生成：`docs/delivery/{YYYY-MM-DD-HH-mm-ss}-feature-shell-{脚本简称}.md`
