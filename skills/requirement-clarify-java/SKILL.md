# 需求澄清协议（Requirement Clarification Protocol）

> **本协议被以下 Skill 引用**：`feature-dev-java`、`bug-fix-java`、`refactor-java`、`code-review-java`、`deploy-doc-java`、`api-doc-java`、`testing-java`
> 
> 当上述 Skill 被触发时，**必须先执行本协议**，通过后方可进入各 Skill 的正式执行阶段。

---

## 核心原则

### 🚫 强制门控规则
1. **禁止在必问项未澄清时进入执行阶段**（输出方案、写代码、生成报告等）
2. **禁止"我觉得自己已经理解了"而跳过追问** — 必须逐项对照检查清单
3. **禁止一次性抛出所有问题** — 每轮最多提出 3~5 个最关键的问题
4. **允许合理推断** — 如果用户给的上下文已足够推断出某个答案，可以说"我理解为 XXX，如有偏差请纠正"而非反复追问

### ✅ 智能退出条件
当以下条件**全部满足**时，自动结束澄清流程，进入 Skill 正式执行阶段：
- 所有 🔴 必问项 已有明确答案（用户提供或可合理推断）
- 🟡 选问项 中没有会严重影响交付质量的遗漏

---

## 执行流程

### 步骤 1：信息充分度自检

收到用户请求后，**先在内部（不输出给用户）** 对照下方「问题清单」完成自检：

```
对照当前 Skill 的问题清单：
- 🔴 必问项 1: [已知 / 未知 / 可推断]
- 🔴 必问项 2: [已知 / 未知 / 可推断]
- 🔴 必问项 3: [已知 / 未知 / 可推断]
- 🟡 选问项 1: [已知 / 未知 / 可推断]
- ...
```

### 步骤 2：结构化追问

将**未知的必问项** + **可能影响质量的未知选问项**组合为一轮追问（≤ 5 个问题），以清晰的编号列表形式呈现：

```markdown
在开始之前，我需要确认几个关键信息：

1. **[问题标题]**: [具体问题描述]
2. **[问题标题]**: [具体问题描述]
3. **[问题标题]**: [具体问题描述]

> 💡 已理解的信息：[简要列出已从用户描述中获取的信息，让用户知道你已理解了什么]
```

### 步骤 3：循环确认

- 用户回答后，更新自检清单
- 如果仍有关键未知项，进行下一轮追问（最多 3 轮，避免过度打扰）
- 如果 3 轮后仍有少量模糊点，可以带着假设推进并在方案中标注"此处基于假设：XXX"

### 步骤 4：退出澄清，进入执行

所有必问项确认完毕后：
1. 输出一段简短的**需求理解确认**（用自己的话复述需求核心）
2. 进入对应 Skill 的正式执行阶段

## 各 Skill 专属问题清单

### 📦 feature-dev（功能开发）

#### 🔴 必问项
| 编号 | 问题 | 说明 |
|:---:|------|------|
| F-01 | **业务目标** | 这个功能要解决什么问题？给谁用？ |
| F-02 | **影响模块** | 涉及哪些现有模块？是新增模块还是在现有模块上扩展？ |
| F-03 | **接口形态** | 是 REST API、定时任务、消息消费者，还是内部方法？ |
| F-04 | **核心业务规则** | 有没有关键的校验规则、状态流转、计算逻辑需要明确？ |

#### 🟡 选问项
| 编号 | 问题 | 说明 |
|:---:|------|------|
| F-05 | **权限要求** | 哪些角色可以操作？是否需要特殊权限？ |
| F-06 | **性能要求** | 有没有并发量、响应时间的约束？ |
| F-07 | **数据要求** | 字段精度、长度限制、唯一性约束？ |
| F-08 | **参考设计** | 有没有类似功能可以参考？有没有设计文档或原型？ |
| F-09 | **异常场景** | 失败时怎么处理？需要重试还是告警？ |
| F-10 | **与其他系统的交互** | 是否需要调用外部服务或被外部服务调用？ |

### 🐛 bug-fix（Bug 修复）

#### 🔴 必问项
| 编号 | 问题 | 说明 |
|:---:|------|------|
| B-01 | **问题现象** | 具体表现是什么？报错信息、异常行为、不符合预期的输出？ |
| B-02 | **复现条件** | 什么操作、什么数据、什么环境下能复现？是必现还是偶现？ |
| B-03 | **期望行为** | 正确的情况应该是怎样的？ |

#### 🟡 选问项
| 编号 | 问题 | 说明 |
|:---:|------|------|
| B-04 | **影响范围** | 是个别用户/数据还是全量影响？ |
| B-05 | **紧急程度** | 是否阻断主流程？需不需要紧急修复？ |
| B-06 | **环境差异** | 测试环境能复现吗？是否只在特定环境出现？ |
| B-07 | **最近变更** | 最近有没有相关代码/配置改动可能引入了这个问题？ |
| B-08 | **日志/截图** | 有没有相关的错误日志或截图？ |

### ♻️ refactor（代码重构）

#### 🔴 必问项
| 编号 | 问题 | 说明 |
|:---:|------|------|
| RF-01 | **重构目标** | 为什么要重构？是为了解耦、提取公共逻辑、提高清晰度、降低复杂度还是便于测试？ |
| RF-02 | **重构范围** | 具体涉及哪些类、包、模块？是否限制只改某几个文件？ |
| RF-03 | **行为约束** | 是否要求外部行为完全不变？允许顺带修复小问题或调整接口吗？ |

#### 🟡 选问项
| 编号 | 问题 | 说明 |
|:---:|------|------|
| RF-04 | **回归保护** | 当前有没有现成测试、接口用例或可用于校验行为一致性的手段？ |
| RF-05 | **性能与事务约束** | 是否有性能、事务、并发或数据库访问路径不能改变的限制？ |
| RF-06 | **禁止改动项** | 哪些公共类、配置、接口契约、SQL、表结构不允许动？ |
| RF-07 | **期望结果** | 希望重构后达到什么状态，如方法拆分、类职责收敛、重复代码消除、圈复杂度下降？ |

## 追问话术规范

### ✅ 好的追问方式
- "在开始之前，我需要确认几个关键信息：1. ... 2. ... 3. ..."
- "我已经理解了 [XXX部分]，但以下几点需要明确：..."
- "关于 XXX，我理解为 [你的推断]，如有偏差请纠正。另外还需确认：..."

### ❌ 不好的追问方式
- ~~"请提供完整的需求文档"~~ → 用户找你就是因为他不想写长文档
- ~~一次列出 10 个以上问题~~ → 信息轰炸，用户会失去耐心
- ~~无实质内容的宽泛问题（"你还有什么补充吗？"）~~ → 要问具体的、有明确选项的问题

---

## 特殊场景处理

### 用户明确说"别问了直接做"
- 尊重用户意愿，直接进入执行阶段
- 但在方案/代码中**标注所有基于假设的决策**，方便用户事后审查

### 用户给的需求已经非常详尽
- 步骤 1 自检通过后，**不需要强行追问**
- 直接输出"需求理解确认"后进入执行

### 紧急修复场景
- 如果用户表达了紧急性（"线上出问题了"、"紧急修复"），缩减追问到最核心的 1~2 个问题
- 优先行动，边做边确认细节
