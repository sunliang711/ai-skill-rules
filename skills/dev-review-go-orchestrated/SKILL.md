# Go 多 Agent 开发 + 评审组合流程

> 当用户的请求同时包含“开发/修复”“做完后 review”“拆子任务并行处理”等意图时使用。
>
> 若运行环境不支持 subagent / delegation，或任务规模较小，则回退到 `dev-review-go`。

## 流程概览

```text
判断任务类型
-> feature-dev-go-orchestrated 或 bug-fix-go
-> review subtask
-> 主 Agent 修复问题
-> doc subtask
-> 合并交付
```

## 阶段一：先决定编排路径

### 步骤 1：识别任务类型

- 新增功能、接口、任务能力：优先执行 `feature-dev-go-orchestrated`
- 修复错误、panic、异常行为：执行 `bug-fix-go`
- 若修复任务本身也需要拆子任务，可沿用本 Skill 中的评审与文档编排规则

### 步骤 2：先做需求澄清

无论开发还是修复，都必须先执行 `requirement-clarify-go`。

至少确认：
- 业务背景
- 影响范围
- 变更类型
- review 关注点

## 阶段二：先完成开发或修复

### 步骤 3：执行主实现流程

不得跳过以下环节：
- 方案设计或定位结论
- 文件边界拆分
- 编码实现
- 测试或验证
- 变更摘要

### 步骤 4：收集评审输入

进入评审前，主 Agent 需要整理：
- 本次新增和修改的文件
- 业务背景
- 变更类型
- 测试结果
- 已知风险和暂未覆盖点

## 阶段三：独立评审

### 步骤 5：跳过重复澄清

开发或修复阶段已经明确的内容，可直接作为评审输入：
- 审查范围：本次变更文件
- 业务背景：阶段一已确认内容
- 变更类型：新功能或 Bug 修复

### 步骤 6：分配 review subtask

review subagent 必须独立于实现 subtask，避免“自己审自己”。

review 输入至少包含：

```markdown
## Review Brief
- 审查范围：
- 业务背景：
- 变更类型：
- 测试结果：
- 已知风险：
- 重点关注：
```

评审时优先检查：
- 是否存在行为错误或遗漏分支
- 是否存在并发、资源释放、context 传递问题
- 是否存在日志泄露、硬编码凭据、注入等安全问题
- 是否缺少必要测试

### 步骤 7：按评审结果处理

- 有阻断问题：主 Agent 优先修复，再进行复审
- 只有警告或建议：输出结论并说明风险接受情况
- 无问题：输出通过结论和残余风险

## 阶段四：文档与最终交付

### 步骤 8：分配 doc subtask

在实现和评审结论稳定后，再交给 doc subagent 整理任务交付文档。

doc 输入至少包含：
- 任务背景
- 实现或修复方案
- 最终文件变更
- 测试结果
- 评审问题清单及处理结果

### 步骤 9：主 Agent 最终汇总

主 Agent 在交付前必须确认：
- 阻断问题已全部处理
- 文档与最终代码一致
- 仍未处理的建议已标注原因
- 用户可直接看到最终结论

## 最终交付

用户最终应获得：
- 已完成的 Go 代码变更
- 测试结果
- 变更摘要
- 独立代码评审报告
- 风险结论
- 任务交付文档

## 任务交付文档

在 `docs/` 下生成：`docs/{YYYY-MM-DD}-dev-review-go-orchestrated-{功能或问题简称}.md`

建议包含：
- 任务背景
- 编排方案
- 实现或修复方案
- 文件与配置变更
- 测试结果
- 评审问题清单与处理结果
- 风险与后续建议
