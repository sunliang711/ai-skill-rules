# Go 代码评审 SOP

> 当用户提出“review 这段 Go 代码”、“帮我审一下这个 PR”、“看下这个 Go 服务实现有没有问题”等请求时使用。

## 前置：需求澄清

> 必须先执行 `requirement-clarify-go`，至少确认 `RG-01` ~ `RG-02`。

## 步骤 1：准备审查
- 确认审查范围。
- 理解业务背景与变更目的。
- 若用户给的是目录或模块，先列出待审文件。

## 步骤 2：按三维度系统审查

> 审查基线为 `rules/04-go-backend.mdc` 与 `rules/05-go-security.mdc`；若变更涉及 Gin Handler、路由或 HTTP 契约，还需同时参考 `rules/06-go-api-design.mdc`。

### 维度一：安全审查
| 编号 | 检查项 | 说明 |
|:---:|--------|------|
| GS-01 | 硬编码凭据 | Token、密码、密钥、连接串是否硬编码 |
| GS-02 | 日志泄密 | 日志是否输出敏感字段 |
| GS-03 | 输入校验 | Handler 是否正确绑定并校验外部输入 |
| GS-04 | SQL/命令注入 | 原始 SQL、shell 命令、外部 URL 是否存在拼接风险 |
| GS-05 | 鉴权缺失 | 受保护接口是否缺少鉴权或权限判断 |
| GS-06 | 不安全配置 | 是否允许关闭 TLS 校验、信任所有证书等 |
| GS-07 | 金额精度 | 金额是否误用 `float32/float64` |
| GS-08 | 敏感数据生命周期 | 密钥、Token 等是否被长生命周期对象持有或泄漏 |

### 维度二：规范审查
| 编号 | 检查项 | 说明 |
|:---:|--------|------|
| GN-01 | 分层清晰 | Handler / Service / Repository 职责是否混淆 |
| GN-02 | 依赖注入 | 是否通过 Fx / 构造函数注入而非全局变量 |
| GN-03 | DTO 分离 | 是否把 Model 直接暴露给外部响应 |
| GN-04 | 路由规范 | Gin 路径是否符合 REST 约定 |
| GN-05 | 命名规范 | 包、类型、方法、ID/URL/HTTP 大写是否一致 |
| GN-06 | 配置管理 | 是否绕过统一配置入口散落读取配置 |
| GN-07 | Tag 规范 | `json/gorm/mapstructure/binding` 是否一致 |
| GN-08 | 注释质量 | 复杂业务和关键取舍是否有必要说明 |

### 维度三：质量审查
| 编号 | 检查项 | 说明 |
|:---:|--------|------|
| GQ-01 | 错误处理 | `err` 是否被正确处理、包装和返回 |
| GQ-02 | 并发安全 | 是否存在数据竞态、goroutine 泄漏、死锁风险 |
| GQ-03 | context 使用 | 是否正确透传 `context.Context` 和超时 |
| GQ-04 | 资源释放 | DB/Redis/HTTP body/定时器等是否正确关闭 |
| GQ-05 | 事务边界 | 事务放置是否合理，是否遗漏回滚路径 |
| GQ-06 | 性能隐患 | N+1、无界缓存、无分页、循环内远程调用 |
| GQ-07 | 可测试性 | 是否依赖具体实现导致难以 mock |
| GQ-08 | 测试充分性 | 关键路径、错误路径和边界是否有测试 |

## 步骤 3：输出审查报告
优先输出发现的问题，格式如下：

```markdown
## Go 代码评审报告

### 审查文件
- [文件列表]

### 发现的问题
| 序号 | 位置 | 严重等级 | 检查项 | 问题描述 | 修复建议 |
|:---:|------|:---:|:---:|----------|----------|

### 审查总结
- 阻断问题：
- 警告问题：
- 建议问题：

### 总体评价
[简要结论]
```

严重等级建议：
- 阻断：可能导致安全问题、错误结果、panic、数据损坏、明显回归
- 警告：本次迭代建议修复
- 建议：可后续优化

## 步骤 4：若无问题
如果没有发现问题，明确写出：

```markdown
## Go 代码评审报告

### 审查结果
未发现明确问题。

### 残余风险/测试缺口
- [如有]
```

## 步骤 5：任务交付文档
在 `docs/` 下生成：`docs/{YYYY-MM-DD}-review-go-{模块简称}.md`

建议包含：
- 审查范围
- 审查维度与基线
- 问题清单
- 风险结论
- 后续建议
