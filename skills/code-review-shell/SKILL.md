# Shell 代码评审 SOP

> 当用户提出“review 这个 Shell 脚本”、“帮我审一下 Bash 脚本改动”等请求时使用。

## 前置：需求澄清

> 必须先执行 `requirement-clarify-shell`，至少确认 `RS-01` ~ `RS-02`。
> 若脚本会产生写入副作用，审查时必须检查脚本内部是否具备写前前置检查逻辑。

## 步骤 1：准备审查
- 确认审查范围、运行环境和执行权限
- 若用户给的是目录或 PR，先列出待审脚本

## 步骤 2：按三维度系统审查

> 审查基线为 `rules/13-shell-scripting.mdc` 与 `rules/14-shell-security.mdc`。

### 维度一：安全
- 是否存在 `eval`、`bash -c`、`sh -c` 拼接执行风险
- 是否有空变量路径删除、权限升级、下载执行等高风险操作
- 是否泄露密钥、Token、连接串或敏感环境变量
- 临时文件、权限、下载和网络调用是否安全

### 维度二：规范
- shebang、严格模式、函数拆分和参数解析是否清晰
- 变量引用、引号、日志和清理逻辑是否一致
- 兼容性目标是否明确（bash / sh / zsh）

### 维度三：质量
- 错误传播、退出码和 `trap` 是否可靠
- 副作用是否可控，是否提供 dry-run 或保护措施
- 是否有 `shellcheck`、`bats` 或清晰的验证步骤

## 步骤 3：输出报告

```markdown
## Shell 代码评审报告
- 审查文件：
- 发现的问题：
- 严重等级：阻断 / 警告 / 建议
- 总体评价：
```

### 步骤 4：若无问题
明确写出“未发现明确问题”，并补充残余风险或验证缺口。

### 步骤 5：任务交付文档
在 `docs/review/` 下生成：`docs/review/{YYYY-MM-DD}-review-shell-{脚本简称}.md`
