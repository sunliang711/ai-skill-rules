# Rust 需求澄清协议

> 本协议被 `feature-dev-rust`、`bug-fix-rust`、`refactor-rust`、`code-review-rust`、`testing-rust`、`deploy-doc-rust` 引用。

## 核心原则

- 必问项未确认前，禁止直接输出方案、写代码或下结论
- 每轮追问控制在 3~5 个关键问题，避免信息轰炸
- 可以基于已有上下文做合理推断，但必须明确标注假设
- 所有 🔴 必问项确认后，才能进入对应 Skill 的正式阶段

## 执行方式

1. 先在内部对照下方问题清单，判断每项为“已知 / 未知 / 可推断”
2. 组合未知的必问项和关键选问项，向用户发起一轮追问
3. 最多迭代 3 轮；若仍有少量模糊点，带着假设继续并在方案中说明
4. 退出前先用自己的话复述 Rust 任务理解

## 专属问题清单

### feature-dev-rust

#### 🔴 必问项
- FR-01：业务目标与边界是什么
- FR-02：涉及哪些 crate / module / service
- FR-03：入口形态是什么：HTTP、CLI、后台任务还是库能力
- FR-04：关键规则、状态流转、并发/事务要求是什么

#### 🟡 选问项
- FR-05：是否涉及数据库、缓存、消息队列或外部 API
- FR-06：是否有性能、超时、吞吐或内存约束
- FR-07：是否有现成模块或实现可参考

### bug-fix-rust

#### 🔴 必问项
- BR-01：问题现象是什么，是否包含 panic / 报错 / 错误结果
- BR-02：复现步骤与触发条件是什么
- BR-03：期望行为是什么

#### 🟡 选问项
- BR-04：影响范围与紧急程度
- BR-05：最近相关变更和日志线索

### code-review-rust

#### 🔴 必问项
- RR-01：审查范围是哪些文件、模块或 PR
- RR-02：这段 Rust 代码解决的业务背景是什么

#### 🟡 选问项
- RR-03：更关注安全、并发、性能还是规范
- RR-04：是否有已知风险点希望重点审查

### refactor-rust

#### 🔴 必问项
- RFR-01：重构目标是什么
- RFR-02：具体改动范围有哪些 module / trait / impl
- RFR-03：是否要求对外行为完全不变

#### 🟡 选问项
- RFR-04：现有测试和回归保护手段是什么
- RFR-05：哪些接口、配置、协议或数据结构不能动

### deploy-doc-rust

#### 🔴 必问项
- DR-01：涉及哪些 crate、服务、二进制或脚本
- DR-02：目标环境和部署方式是什么

#### 🟡 选问项
- DR-03：是否包含迁移、配置变更、systemd / 容器发布
- DR-04：回滚要求和停机窗口

### testing-rust

#### 🔴 必问项
- TR-01：要测试哪些 module / function / handler
- TR-02：测试类型是单元、集成、HTTP 还是异步场景

#### 🟡 选问项
- TR-03：必须覆盖的错误路径、边界条件或并发场景
- TR-04：外部依赖应 mock、fake 还是走隔离环境
