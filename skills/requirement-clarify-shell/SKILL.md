# Shell 需求澄清协议

> 本协议被 `feature-dev-shell`、`bug-fix-shell`、`refactor-shell`、`code-review-shell`、`testing-shell`、`deploy-doc-shell` 引用。

## 核心原则

- 必问项未确认前，禁止直接写脚本或执行危险命令
- 每轮追问控制在 3~5 个关键问题
- 可合理推断，但必须标注假设
- 所有 🔴 必问项确认后，才能进入正式执行阶段
- 生成或修改会产生写入副作用的脚本时，脚本内部必须在实际写入、删除、移动、部署、权限修改等操作前统一完成前置检查：目标路径、写入权限、依赖命令、运行环境、必要配置和输出目录；若存在缺失项，脚本必须输出明确错误并退出，禁止执行到一半才因依赖不满足失败

## 专属问题清单

### feature-dev-shell

#### 🔴 必问项
- FS-01：脚本要解决什么问题，给谁用
- FS-02：运行环境是什么：bash / sh / zsh，Linux / macOS / CI / 容器
- FS-03：输入、输出、依赖命令和副作用是什么
- FS-04：哪些路径、环境变量、权限或破坏性操作需要特别约束

#### 🟡 选问项
- FS-05：是否需要 dry-run、交互确认或回滚
- FS-06：是否有现成脚本或运维流程可参考

### bug-fix-shell

#### 🔴 必问项
- BS-01：问题现象与报错是什么
- BS-02：复现步骤、输入参数和运行环境是什么
- BS-03：期望行为是什么

#### 🟡 选问项
- BS-04：影响范围与紧急程度
- BS-05：最近变更与执行日志

### code-review-shell

#### 🔴 必问项
- RS-01：审查范围是哪些脚本或变更
- RS-02：脚本的用途、运行环境和执行权限是什么

#### 🟡 选问项
- RS-03：更关注安全、兼容性、可维护性还是可观测性
- RS-04：是否有已知高风险操作

### refactor-shell

#### 🔴 必问项
- RFS-01：重构目标是什么
- RFS-02：具体涉及哪些脚本、函数、命令链
- RFS-03：是否要求对外行为、参数和输出完全不变

#### 🟡 选问项
- RFS-04：现有验证手段与回归保护
- RFS-05：哪些危险命令、路径或权限逻辑不能动

### deploy-doc-shell

#### 🔴 必问项
- DS-01：涉及哪些脚本、定时任务、systemd / CI / cron 配置
- DS-02：目标环境和执行方式是什么

#### 🟡 选问项
- DS-03：是否包含权限变更、路径初始化、依赖安装
- DS-04：回滚与停机要求

### testing-shell

#### 🔴 必问项
- TS-01：要验证哪些脚本、函数或参数分支
- TS-02：测试方式是 shellcheck、bats、手工验证还是组合方式

#### 🟡 选问项
- TS-03：必须覆盖的错误路径和破坏性操作保护
- TS-04：是否需要 mock 外部命令或隔离临时目录
