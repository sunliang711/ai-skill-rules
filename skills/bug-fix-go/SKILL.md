# Go Bug 修复 SOP

> 当用户提出“修复 Go bug”、“排查 panic”、“定位接口异常”、“修一个并发问题”等请求时使用。

## 前置：需求澄清

> 必须先执行 `requirement-clarify-go`，至少确认 `BG-01` ~ `BG-03`。

## 阶段一：问题理解与定位

### 步骤 1：整理已知信息
至少明确：
- 问题现象
- 复现路径
- 期望行为
- 影响范围与紧急程度

### 步骤 2：按 Go 常见根因定位
重点检查以下方向：
- panic / nil pointer / map 并发写
- `context` 超时或取消传播错误
- 错误被吞掉、包装丢失或判断不准确
- goroutine 泄漏、死锁、竞态条件
- 配置项缺失或依赖初始化顺序问题
- 数据库、Redis、MQ 调用的超时与事务问题

### 步骤 3：先输出定位结论
定位后先向用户汇报：

```markdown
## Bug 定位分析

### 问题现象
[复述问题]

### 根因分析
- 根因位置：
- 问题类型：逻辑 / 并发 / 配置 / 依赖 / 数据 / 错误处理
- 触发条件：
- 为什么会发生：

### 修复方案
- 修改文件：
- 修复思路：
- 影响评估：
- 是否需要同步排查同类代码：
```

## 阶段二：修复实施

> 修复时必须遵循 `rules/04-go-backend.mdc`，并坚持最小必要改动；如问题涉及鉴权、密钥、日志泄漏、TLS、注入或配置安全，还必须对照 `rules/05-go-security.mdc`。

### 步骤 4：最小化修复
- 只改与问题直接相关的代码。
- 避免顺手重构和风格化清理。
- 若涉及公共函数、共享结构或配置入口，先评估所有调用方。

### 步骤 5：按问题类型补防御
- 空值问题：补 `nil` 检查和前置校验。
- 错误链问题：补 `%w` 包装和 `errors.Is/As` 可判定性。
- 并发问题：补锁、channel 约束或生命周期控制。
- 超时问题：补 `context`、deadline、资源关闭。
- 配置问题：补默认值、校验和启动期失败保护。

## 阶段三：验证与交付

### 步骤 6：验证修复有效性
- 原问题能复现的，优先复现再验证修复。
- 无法稳定复现的，至少补针对性测试或验证步骤。
- 检查修复是否改变了旧调用方的契约。

### 步骤 7：自检清单

#### 修复有效性
- [ ] 原问题已消除
- [ ] 已覆盖已知触发条件
- [ ] 未破坏现有调用方
- [ ] 已评估相同模式下的同类风险

#### Go 质量
- [ ] `err` 未被忽略
- [ ] 并发修复不会引入新竞态
- [ ] `context` 传递与取消正确
- [ ] 无额外全局状态污染

#### 安全与稳定性
- [ ] 日志未泄漏敏感信息
- [ ] 超时、重试、资源释放合理
- [ ] 配置和依赖初始化顺序正确

### 步骤 8：修复摘要

```markdown
## Bug 修复摘要
- 问题：
- 根因：
- 修复方式：
- 影响范围：
- 验证方式：
- 回归风险：
```

### 步骤 9：ChangeLog 条目

```markdown
## [版本号] - YYYY-MM-DD

### Fixed
- [模块/功能]：修复 [问题描述]，根因为 [简述根因]
```

### 步骤 10：任务交付文档
在 `docs/delivery/` 下生成：`docs/delivery/{YYYY-MM-DD-HH-mm-ss}-bugfix-go-{问题简称}.md`

建议包含：
- 问题背景
- 根因分析
- 修复方案
- 文件与配置变更
- 验证结果
- 风险与后续建议
