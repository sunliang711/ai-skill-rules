# Rust Bug 修复 SOP

> 当用户提出“修复 Rust bug”、“排查 panic”、“定位 async 问题”等请求时使用。

## 前置：需求澄清

> 必须先执行 `requirement-clarify-rust`，至少确认 `BR-01` ~ `BR-03`。

## 阶段一：问题理解与定位

### 步骤 1：整理已知信息
- 问题现象、复现路径、期望行为
- 影响范围与紧急程度
- 是否与 panic、超时、死锁、权限或配置有关

### 步骤 2：按 Rust 常见根因定位
- `unwrap()` / `expect()` / `panic!` 引发异常
- 错误链丢失、Result 处理不完整
- async 任务泄漏、锁使用不当、共享状态竞态
- 配置缺失、路径错误、权限问题
- SQL / 命令调用 / 外部服务异常处理不足

### 步骤 3：先输出定位结论

```markdown
## Bug 定位分析
- 问题现象：
- 根因位置：
- 触发条件：
- 修复思路：
- 影响评估：
```

## 阶段二：修复实施

> 修复时必须遵循 `rules/07-rust-backend.mdc`；涉及安全、命令执行、凭据、注入或 TLS 时，同时遵循 `rules/08-rust-security.mdc`。

### 步骤 4：最小化修复
- 只改与问题直接相关的代码
- 避免顺手重构和风格化清理
- 涉及公共 trait、共享模块或配置入口时，先评估所有调用方

### 步骤 5：按问题类型补防御
- panic：改为显式错误返回或前置校验
- 并发：补锁、超时、取消或生命周期控制
- 配置：补默认值、校验和启动期失败保护
- 外部依赖：补错误包装、重试/降级和资源释放

## 阶段三：验证与交付

### 步骤 6：验证修复有效性
- 能复现的先复现再验证
- 不能稳定复现的，至少补针对性测试或验证步骤
- 检查修复是否改变旧接口、命令或协议行为

### 步骤 7：输出结果

```markdown
## Bug 修复摘要
- 问题：
- 根因：
- 修复方式：
- 影响范围：
- 验证方式：
- 回归风险：
```

### 步骤 8：任务交付文档
在 `docs/delivery/` 下生成：`docs/delivery/{YYYY-MM-DD}-bugfix-rust-{问题简称}.md`
