# Python Bug 修复 SOP

> 当用户提出“修复 Python bug”、“排查 FastAPI 异常”、“定位脚本报错”等请求时使用。

## 前置：需求澄清

> 必须先执行 `requirement-clarify-python`，至少确认 `BP-01` ~ `BP-03`。

## 阶段一：问题理解与定位

### 步骤 1：整理已知信息
- 问题现象、复现步骤、期望行为
- 影响范围、紧急程度、环境差异
- 是否与配置、权限、并发、任务调度或外部依赖相关

### 步骤 2：按 Python 常见根因定位
- 空值、KeyError、类型不匹配、序列化问题
- 异常被吞掉、错误包装不当、返回码错误
- 会话、文件、连接、任务或线程未正确关闭
- 配置缺失、环境变量问题、路径错误
- SQL / shell / URL 拼接带来的注入或执行问题

### 步骤 3：先输出定位结论

```markdown
## Bug 定位分析
- 问题现象：
- 根因位置：
- 触发条件：
- 修复思路：
- 影响评估：
```

## 阶段二：修复实施

> 修复时必须遵循 `rules/10-python-backend.mdc`；涉及安全能力时，同时遵循 `rules/11-python-security.mdc`。

### 步骤 4：最小化修复
- 只改与问题直接相关的代码
- 避免顺手重构
- 涉及公共函数、配置入口或协议结构时，先评估所有调用方

### 步骤 5：补防御措施
- 参数/空值问题：补校验和默认值
- 异常问题：补错误链和统一返回
- 资源问题：补上下文管理和关闭逻辑
- 并发/任务问题：补超时、取消和幂等保护

## 阶段三：验证与交付

### 步骤 6：验证修复有效性
- 能复现的先复现再验证
- 不能稳定复现的，至少补测试或手工验证步骤
- 检查是否破坏旧接口、脚本参数或任务行为

### 步骤 7：输出结果

```markdown
## Bug 修复摘要
- 问题：
- 根因：
- 修复方式：
- 影响范围：
- 验证方式：
- 回归风险：
```

### 步骤 8：任务交付文档
在 `docs/delivery/` 下生成：`docs/delivery/{YYYY-MM-DD}-bugfix-python-{问题简称}.md`
