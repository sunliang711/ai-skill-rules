# Java 代码重构 SOP

> 当用户提出“重构这段 Java 代码”、“优化这个类结构”、“抽公共方法”、“拆分 Service/Controller 逻辑”等请求时使用。

## 前置：需求澄清

> 必须先执行 `requirement-clarify-java`，至少确认 `RF-01` ~ `RF-03`。

## 核心原则

- 默认目标是**不改变外部行为**，除非用户明确允许行为调整。
- 重构与新功能开发分开处理，禁止借重构顺手新增需求。
- 重构与 Bug 修复分开说明；如发现缺陷，先告知用户是否一并处理。
- 优先小步快跑，避免一次性大改导致回归面失控。

## 阶段一：重构分析与方案

### 步骤 1：识别重构动机
明确当前代码的主要问题，例如：
- 类职责过多
- 方法过长或嵌套过深
- 重复逻辑分散
- 层次边界混乱
- 可测试性差
- 命名含义不清

### 步骤 2：识别行为边界
在动手前明确哪些内容必须保持不变：
- 对外接口契约
- SQL 与事务语义
- 权限、校验与异常码
- 日志与审计关键点
- 配置项名称和默认值

### 步骤 3：先输出重构方案
先给出方案，再等待用户确认：

```markdown
## 重构方案

### 1. 重构目标
- [本次要解决的结构性问题]

### 2. 重构范围
- 涉及文件：
- 涉及模块：
- 明确不改动的部分：

### 3. 行为保持策略
- 对外接口：
- 数据库与事务：
- 权限与异常：

### 4. 拆分步骤
1. [步骤一]
2. [步骤二]
3. [步骤三]

### 5. 回归验证
- 现有测试：
- 计划补充的测试：
- 人工验证点：
```

## 阶段二：实施重构

> 编码时遵循 `rules/01-java-backend.mdc`、`rules/02-java-security.mdc`，涉及 Controller/API 时再叠加 `rules/03-java-api-design.mdc`。

### 步骤 4：按最小风险顺序实施
推荐顺序：
1. 先提取常量、私有方法、纯工具逻辑
2. 再拆分 Service / Repository / DTO 边界
3. 最后处理对外接口附近的结构调整

### 步骤 5：重构约束
- 保持方法签名、返回结构和异常语义稳定，除非用户明确允许调整
- 不修改无关配置、依赖版本或公共基础设施
- 不在重构过程中改变权限判断、加密方式、日志脱敏策略
- 拆分类和方法时，优先保持命名、分层和职责一致性

### 步骤 6：必要时补保护性测试
- 若已有测试不足以覆盖行为边界，先补关键回归测试再继续重构
- 至少保护以下内容：
  - 核心业务分支
  - 边界条件
  - 异常分支
  - 对外接口关键响应

## 阶段三：验证与交付

### 步骤 7：自检清单

#### 行为一致性
- [ ] 对外接口契约未破坏
- [ ] 业务规则与异常语义未改变
- [ ] 权限、校验、日志与审计关键点未丢失
- [ ] 数据库访问与事务边界未被误改

#### 结构改进
- [ ] 类职责更清晰
- [ ] 方法复杂度下降
- [ ] 重复逻辑已收敛
- [ ] 命名更准确一致

#### 风险控制
- [ ] 改动范围聚焦
- [ ] 无顺手新增需求
- [ ] 相关测试已运行或给出验证说明

### 步骤 8：输出重构摘要

```markdown
## 重构摘要
- 重构目标：
- 涉及文件：
- 主要结构调整：
- 保持不变的行为：
- 验证方式：
- 回归风险：
```

### 步骤 9：任务交付文档
在 `docs/` 下生成：`docs/{YYYY-MM-DD}-refactor-java-{模块简称}.md`

建议包含：
- 重构背景
- 重构方案
- 文件变更
- 行为保持策略
- 测试与验证结果
- 风险与后续建议
