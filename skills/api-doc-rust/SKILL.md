# Rust API 文档 SOP

> 当用户提出“生成 Rust API 文档”、“整理 HTTP 接口说明”、“补 curl 示例”等请求时使用。

## 前置：需求澄清

> 必须先执行 `requirement-clarify-rust`，至少确认 `AR-01` ~ `AR-03`。

## 步骤 1：确认文档范围
- 涉及哪些 handler、router、module 或服务
- 文档面向谁使用，是否用于对外开放或内部联调
- 输出 Markdown、OpenAPI 说明，还是同时维护两者

重点扫描：
- `handler/`、`router/`、`transport/http/`
- 请求 / 响应结构体
- 鉴权中间件、错误类型、统一响应与日志追踪
- 现有 utoipa / aide / OpenAPI 文档配置

## 步骤 2：提取接口契约
- 方法、路径、版本号、用途
- 路径参数、查询参数、Header、Body 字段及约束
- 成功响应、错误响应、状态码和业务错误码
- 鉴权、分页、幂等、超时、限流、审计要求
- `curl` 示例、请求示例、响应示例

## 步骤 4：同步现有文档机制
- 若项目已使用 `utoipa`、`aide` 或其他 OpenAPI 方案，新增接口需同步更新
- 若没有自动文档机制，至少生成可联调的 Markdown 文档
- 若用户要求 spec 文件，优先沿用现有框架能力，不手写脱离实现的定义

## 步骤 5：交叉验证
- 路由、方法、参数结构与 handler / router 实现一致
- 错误码与错误类型映射一致
- `curl` 示例与鉴权要求一致，可直接用于联调
- 示例值已脱敏，未暴露内部地址、Token、密钥或用户敏感数据

## 步骤 6：任务交付文档
在 `docs/api/` 下生成：`docs/api/{YYYY-MM-DD}-api-doc-rust-{模块简称}.md`
