# Rust 功能开发 SOP

> 当用户提出“开发 Rust 功能”、“实现 Rust 接口”、“补一个 Rust CLI/后台任务”等请求时使用。

## 前置：需求澄清

> 必须先执行 `requirement-clarify-rust`，至少确认 `FR-01` ~ `FR-04`。

## 阶段一：理解需求与设计方案

### 步骤 1：整理已知信息
- 业务目标、边界和关键规则
- 影响的 crate / module / entrypoint
- 外部依赖、数据模型和并发/事务要求

### 步骤 2：先输出实现方案
编码前先给出方案，并等待用户确认：

```markdown
## 实现方案

### 1. 影响范围
- 涉及 crate / module：
- 新增文件：
- 修改文件：

### 2. 入口与接口
- 入口类型：HTTP / CLI / Worker / Library
- 路由、命令或调用入口：
- 请求与响应结构：

### 3. 数据与依赖变更
- 数据库 / 缓存 / MQ：
- Cargo 依赖变更：
- 配置项变更：

### 4. 核心逻辑
- 处理流程：
- 错误处理：
- 并发 / 超时 / 重试：
```

### 步骤 3：等待用户确认
- 用户确认后再编码
- 如范围变化，先更新方案再继续

## 阶段二：编码实现

> 编码时必须遵循 `rules/07-rust-backend.mdc`；涉及密钥、鉴权、命令调用、注入或 TLS 时，还必须遵循 `rules/08-rust-security.mdc`；涉及 HTTP API 时，再叠加 `rules/09-rust-api-design.mdc`。

### 步骤 4：按推荐顺序实现
- 先补配置与常量，禁止把配置读取散落在业务逻辑中
- 再定义请求/响应 DTO 与领域模型，禁止把持久化模型直接透出
- Repository 负责数据访问，Service 负责业务逻辑与事务/协调
- Handler / Router / CLI 入口只做参数提取、校验和响应转换
- 异步任务、共享状态和后台 `spawn` 必须明确生命周期和取消策略
- 禁止随意使用 `unwrap()` / `expect()` 处理可恢复错误

## 阶段三：测试、自检与交付

### 步骤 5：测试要求
- 为新增或改动的核心逻辑补测试
- 覆盖正常路径、错误路径、边界条件和异步/并发场景（如适用）
- 外部依赖优先 mock、fake 或隔离环境
- 至少说明应执行的检查，如 `cargo test`、`cargo clippy`、`cargo fmt --check`

### 步骤 6：自检清单
- [ ] 分层边界清晰，入口层无业务逻辑
- [ ] 可恢复错误未被 `unwrap()/expect()` 吞掉
- [ ] 并发与后台任务有退出和超时控制
- [ ] 日志无敏感信息
- [ ] 配置、凭据、路径未硬编码
- [ ] 测试覆盖核心路径并可运行

### 步骤 7：输出交付摘要

```markdown
## 变更摘要
- 新增文件：
- 修改文件：
- 新增接口 / 命令 / 任务：
- 配置或依赖变更：
- 测试情况：
```

### 步骤 8：任务交付文档
在 `docs/` 下生成：`docs/{YYYY-MM-DD}-feature-rust-{功能简称}.md`
