# Go 多 Agent 功能开发流程

> 当用户提出“这个 Go 功能请拆子任务处理”“并行做实现、review、文档”“让 subagent 分工协作”等请求时使用。
>
> 若运行环境不支持 subagent / delegation，或任务规模很小，则回退到 `feature-dev-go`。

## 前置：需求澄清

> 必须先执行 `requirement-clarify-go`，至少确认 `FG-01` ~ `FG-04` 后再进入任务拆分。

在以下信息未明确前，禁止拆分 subtask：
- 业务目标与边界
- 影响的包、分层和入口形态
- 核心规则、异常分支、并发与事务要求
- 是否要求额外交付 code review、测试补充或任务文档

## 阶段一：判断是否启用多 Agent

### 步骤 1：只在合适场景启用

满足以下任意两项时，优先启用本流程：
- 影响 3 个及以上文件，或同时影响 2 层及以上分层
- 需求天然可拆为代码调研、实现、评审、文档等独立子任务
- 用户明确要求拆子任务或并行处理
- 除编码外，还要求 review、文档、测试补充等多阶段交付

以下场景不建议启用：
- 单文件或局部小改动
- 需求尚未稳定，方案仍在频繁变化
- 多个子任务无法划清可写文件边界

### 步骤 2：先输出拆分方案并等待确认

在实际调度前，主 Agent 先给出任务拆分方案，并等待用户确认：

```markdown
## 多 Agent 实现方案

### 1. 目标与范围
- 业务目标：
- 本次边界：
- 不在本次范围内：

### 2. 子任务拆分

#### T1 代码调研
- 角色：explorer
- 目标：
- 只读范围：
- 输出物：

#### T2 核心实现
- 角色：worker
- 目标：
- 可写范围：
- 禁止修改：
- 输出物：

#### T3 独立评审
- 角色：review
- 目标：
- 审查范围：
- 输出物：

#### T4 任务文档
- 角色：doc
- 目标：
- 可写范围：docs/
- 输出物：

### 3. 风险与依赖
- 并发 / 事务：
- 外部依赖：
- 配置或数据变更：
```

若用户修改拆分边界，先更新方案，再进入执行。

## 阶段二：定义主 Agent 与 subagent 边界

### 步骤 3：主 Agent 固定负责

主 Agent 始终负责以下事项，不得整体下放：
- 需求澄清与需求理解确认
- 实现方案与任务拆分
- 子任务输入整理与边界约束
- 多个子任务结果集成
- 最终对用户答复

### 步骤 4：各类 subagent 职责

#### 4.1 explorer subagent
- 阅读代码、梳理现状、确认影响范围
- 查找可复用实现、公共组件和相似案例
- 输出风险点、依赖点和建议切入位置
- 不直接修改代码

#### 4.2 worker subagent
- 在明确可写边界内实现功能
- 补充对应测试或验证逻辑
- 返回修改文件、测试情况和残余风险
- 不得越权修改未分配文件

#### 4.3 review subagent
- 仅审查本次变更文件
- 优先关注行为错误、并发风险、安全问题、测试缺口
- 输出阻断问题、一般建议和残余风险
- 不直接重构无关代码

#### 4.4 doc subagent
- 基于最终实现结果整理任务交付文档
- 只写已落地内容，不发明不存在的实现细节
- 文档路径固定在 `docs/`

## 阶段三：拆分与执行

### 步骤 5：推荐执行顺序

```text
主 Agent 完成需求澄清与方案
-> explorer 调研现状
-> 主 Agent 细化文件边界
-> worker 实现代码
-> 主 Agent 集成并补漏
-> review 独立审查
-> 主 Agent 修复阻断问题
-> doc 生成任务文档
-> 主 Agent 汇总交付
```

### 步骤 6：并行规则

允许并行：
- 代码调研与文档模板预填
- 多个 worker 处理不同文件边界
- review 在实现稳定后独立执行

禁止并行：
- 多个 worker 同时修改同一文件
- 需求未定时提前实现
- 代码仍在频繁变化时产出最终评审结论
- 文档在实现未落定前生成最终版

### 步骤 7：子任务输入契约

主 Agent 给 subagent 的输入至少包含：
- 子任务编号与目标
- 背景与业务边界
- 明确的只读 / 可写范围
- 必须遵循的 Go 规则与安全约束
- 预期输出格式

推荐使用以下模板：

```markdown
## Subtask Brief
- 任务编号：
- 子任务类型：explorer / worker / review / doc
- 背景：
- 目标：
- 只读范围：
- 可写范围：
- 禁止修改：
- 必须关注：
- 输出格式：
```

### 步骤 8：子任务输出契约

每个 subagent 返回时必须包含：

```markdown
## Subtask Result
- 任务编号：
- 是否完成：
- 涉及文件：
- 关键结论：
- 测试或验证情况：
- 风险 / 阻塞：
- 建议下一步：
```

## 阶段四：实现与集成要求

### 步骤 9：编码仍遵循 Go 基础 Skill

即使启用多 Agent，编码实现仍必须遵循：
- `feature-dev-go`
- `rules/04-go-backend.mdc`
- 涉及安全时叠加 `rules/05-go-security.mdc`
- 涉及 HTTP 接口时叠加 `rules/06-go-api-design.mdc`

本 Skill 只负责“拆分与编排”，不替代底层 Go 编码规范。

### 步骤 10：主 Agent 集成检查

主 Agent 在最终交付前必须确认：
- 子任务输出之间无冲突
- 实现与最初方案一致
- review 阻断项已处理
- 文档内容与最终代码一致
- 测试结果与未验证部分已明确说明

## 阶段五：最终交付

### 步骤 11：交付内容

最终至少包含：
- 变更摘要
- 文件范围
- 测试结果
- review 结论
- 风险与后续建议

### 步骤 12：任务交付文档

在 `docs/` 下生成：`docs/{YYYY-MM-DD}-feature-go-orchestrated-{功能简称}.md`

建议包含：
- 任务背景
- 多 Agent 拆分方案
- 各子任务输出摘要
- 最终文件变更
- 配置与依赖变更
- 测试结果
- 评审结论
- 风险与后续建议
