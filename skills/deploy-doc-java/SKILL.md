# 部署文档生成 SOP（Standard Operating Procedure）

> 当用户提出"生成部署文档"、"这次改动需要什么部署操作"、"列出新增的配置项"等部署相关请求时，严格按以下流程执行。

---

## 前置：需求澄清（强制）

> ⚠️ **必须先执行 `requirement-clarify-java` Skill 中的澄清协议，通过后方可进入正式执行流程。**
> 
> 对照 `requirement-clarify-java` 中「deploy-doc 专属问题清单」的 🔴 必问项（D-01 ~ D-02）进行信息充分度自检，
> 缺失项必须向用户追问，**禁止跳过此环节直接生成文档**。

---

## 步骤 1：确定变更范围

### 1.1 识别变更文件
向用户确认本次变更涉及的范围，或通过以下方式自动识别：
- 用户指定的文件列表
- Git diff（如可获取）
- 用户在本次对话中修改过的所有文件

### 1.2 重点扫描目标
以下类型文件是部署文档的**核心信息来源**：

| 文件类型 | 扫描目标 |
|----------|----------|
| `application*.yml` / `application*.properties` | 新增/修改的配置项、`${ENV_VAR}` 占位符 |
| `pom.xml` / `build.gradle` | 新增/升级/删除的依赖 |
| `*.sql` / Entity 类 | 数据库表结构变更（新增表、新增字段、新增索引） |
| `Dockerfile` / `docker-compose.yml` | 容器配置变更（端口、卷挂载、环境变量） |
| 脚本文件（`*.sh` / `*.bat`） | 部署脚本变更 |
| `.env` / `.env.example` / `.secret` | 环境变量模板变更 |
| `logback*.xml` / `log4j2*.xml` | 日志配置变更 |
| `*.pem` / `*.p12` / `*.jks` / `truststore` | 证书/密钥文件变更（mTLS、SSL） |
| Nginx / Gateway 配置 | 路由规则变更 |

---

## 步骤 2：逐项提取变更信息

### 2.1 环境变量提取
扫描所有 `${...}` 占位符，提取出需要在部署环境中配置的环境变量：

需要记录的信息：
- 变量名（精确拼写）
- 用途说明
- 示例值
- 是否敏感（密码/密钥/Token 标记为敏感，部署时应通过密钥管理系统注入）
- 影响的服务/模块

### 2.2 配置项提取
扫描 `application*.yml` 中新增的非环境变量配置项：

需要记录的信息：
- 完整配置路径（如 `sms.expire-seconds`）
- 当前值/默认值
- 用途说明
- 是否需要按环境调整

### 2.3 数据库变更提取
扫描 SQL 脚本或 Entity 类变更：

需要记录的信息：
- 变更类型（建表 / 加字段 / 加索引 / 修改字段）
- 完整的 DDL 语句
- 执行顺序（是否有依赖关系）
- 是否需要数据迁移
- 是否可回滚

### 2.4 依赖变更提取
对比 `pom.xml` 或 `build.gradle` 变更：

需要记录的信息：
- 依赖坐标（GroupId:ArtifactId）
- 旧版本 → 新版本（新增则标记为"新增"）
- 变更原因

### 2.5 中间件与基础设施变更
识别新增的中间件依赖：

- Redis: 新增的 Key 命名规则与用途
- MQ: 新增的 Topic / Queue / Exchange
- Elasticsearch: 新增的索引与 Mapping
- 新增的服务端口
- 新增的文件存储路径

### 2.6 证书与密钥文件变更
识别是否涉及证书、密钥文件的新增或更换：

- SSL/TLS 证书（服务端证书、CA 证书）
- mTLS 客户端证书
- Keystore / Truststore 文件
- Vault/KMS 相关配置变更

## 三、数据库变更

### 执行顺序
1. 先执行建表脚本
2. 再执行索引脚本

### DDL 脚本
```sql
-- 1. 新建 sms_verify_code 表
CREATE TABLE sms_verify_code (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    phone VARCHAR(20) NOT NULL,
    code VARCHAR(6) NOT NULL,
    expire_time DATETIME NOT NULL,
    used TINYINT DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 2. 添加索引
CREATE INDEX idx_phone_expire ON sms_verify_code(phone, expire_time);
```

### 回滚脚本
```sql
DROP TABLE IF EXISTS sms_verify_code;
```

## 六、部署步骤

> 按以下顺序执行，不可跳过或调换顺序。

1. **数据库变更**
   - 在目标数据库执行 DDL 脚本（见第三章）
   - 确认表和索引创建成功

2. **环境变量配置**
   - 在部署环境中配置第一章列出的所有环境变量
   - 敏感变量通过密钥管理系统注入

3. **中间件准备**
   - 确认 Redis 实例可用且连通（见第五章）

4. **服务部署**
   - 部署新版本 JAR/镜像
   - 启动服务并观察启动日志

5. **验证清单**
   - [ ] 服务启动无报错
   - [ ] 健康检查 `/actuator/health` 返回 UP
   - [ ] 核心接口冒烟测试通过
   - [ ] 日志级别和格式正确

---

## 七、回滚方案

> 如部署后出现严重问题，按以下步骤回滚。

1. 停止新版本服务
2. 回退 JAR/镜像至上一版本
3. 执行数据库回滚脚本（见第三章）
4. 移除新增的环境变量（可选，不影响旧版本运行）
5. 重启服务并验证

## 步骤 4：交叉验证

文档生成后，进行以下验证：
1. **配置完整性**: 逐个核对代码中所有 `${ENV_VAR}` 是否都在文档中列出
2. **DDL 正确性**: 确认 SQL 语法正确、字段类型与 Entity 一致
3. **依赖一致性**: 确认 pom.xml 中实际变更与文档记录一致
4. **顺序合理性**: 确认部署步骤的先后顺序不会导致依赖缺失
5. **安全配置**: 确认 SSL/TLS、Vault、证书等安全相关配置已在文档中列出，敏感变量已标记

验证通过后，向用户交付最终文档。
