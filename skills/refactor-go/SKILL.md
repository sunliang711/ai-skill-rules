# Go 代码重构 SOP

> 当用户提出“重构这段 Go 代码”、“优化包结构”、“拆分 Handler/Service/Repo”、“抽公共逻辑”、“收敛接口依赖”等请求时使用。

## 前置：需求澄清

> 必须先执行 `requirement-clarify-go`，至少确认 `RFG-01` ~ `RFG-03`。

## 核心原则

- 默认目标是**不改变外部行为**，除非用户明确允许行为调整。
- 重构与新功能开发分开处理，禁止借重构顺手新增需求。
- 发现 Bug 时先告知用户是否一并处理，不擅自改变行为。
- 优先做可回滚的小步改动，避免一次性重排多个包或依赖关系。

## 阶段一：重构分析与方案

### 步骤 1：识别结构性问题
重点识别：
- 包职责混乱
- Handler / Service / Repo 边界不清
- 方法过长或嵌套过深
- 重复逻辑散落
- 依赖方向不合理
- 接口抽象不足，难以测试
- `context`、并发控制或错误处理分散混乱

### 步骤 2：识别行为边界
明确哪些内容必须保持不变：
- HTTP / gRPC / 内部接口契约
- 路由、状态码、响应体
- 数据访问路径与事务语义
- 配置结构与注入方式
- 并发行为、锁语义、goroutine 生命周期
- 错误链、日志字段、traceID 传播

### 步骤 3：先输出重构方案

```markdown

### 1. 重构目标
- [本次要解决的结构性问题]

### 2. 重构范围
- 涉及包：
- 涉及文件：
- 明确不改动的部分：

### 3. 行为保持策略
- 接口契约：
- 数据与事务：
- 并发与 context：
- 配置与依赖注入：

### 4. 拆分步骤
1. [步骤一]
2. [步骤二]
3. [步骤三]

### 5. 回归验证
- 现有测试：
- 计划补充的测试：
- 人工验证点：
```

## 阶段二：实施重构

> 编码时遵循 `rules/04-go-backend.mdc`，涉及安全能力时同时遵循 `rules/05-go-security.mdc`，涉及 Handler / API 时再叠加 `rules/06-go-api-design.mdc`。

### 步骤 4：按低风险顺序实施
推荐顺序：
1. 先提取常量、私有函数、纯逻辑辅助函数
2. 再收敛接口与依赖注入边界
3. 再拆分包、结构体职责和公共组件
4. 最后处理路由、Handler 或对外入口附近的调整

### 步骤 5：重构约束
- 保持现有函数行为、错误语义和对外契约稳定
- 不擅自更改配置源、路由路径、中间件顺序和依赖初始化方式
- 不破坏 `context.Context` 透传、traceID 传播和资源释放逻辑
- 不引入新的 goroutine 泄漏、竞态条件或锁顺序风险

### 步骤 6：必要时补保护性测试
- 若现有测试不足，先补关键回归测试
- 优先补：
  - 表驱动单测
  - Handler 响应断言
  - 错误链断言
  - 并发/超时关键路径验证

## 阶段三：验证与交付

### 步骤 7：自检清单

#### 行为一致性
- [ ] 对外接口契约未破坏
- [ ] 路由、状态码、响应体语义未误改
- [ ] `context` 透传和取消行为正确
- [ ] 错误链、日志与追踪字段未丢失
- [ ] 数据访问和事务语义未误改

#### 结构改进
- [ ] 包职责更清晰
- [ ] 依赖方向更合理
- [ ] 重复逻辑已收敛
- [ ] 可测试性提升

#### 风险控制
- [ ] 改动范围聚焦
- [ ] 无顺手新增需求
- [ ] 并发与资源释放风险已评估
- [ ] 相关测试已运行或给出验证说明

### 步骤 8：输出重构摘要

```markdown
## 重构摘要
- 重构目标：
- 涉及包/文件：
- 主要结构调整：
- 保持不变的行为：
- 验证方式：
- 回归风险：
```

### 步骤 9：任务交付文档
在 `docs/delivery/` 下生成：`docs/delivery/{YYYY-MM-DD-HH-mm-ss}-refactor-go-{模块简称}.md`

建议包含：
- 重构背景
- 重构方案
- 文件变更
- 行为保持策略
- 测试与验证结果
- 风险与后续建议
