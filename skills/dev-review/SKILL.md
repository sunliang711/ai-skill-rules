# 开发 + 评审组合流程

> 当用户的请求同时包含**开发/修复**与**代码评审**意图时触发本流程。
> 典型触发语句："开发XXX功能，完成后帮我 review"、"修复这个 bug 并做 code review"、"做完帮我审查一下代码"。

---

## 流程概览

```
┌─────────────┐     ┌──────────────────┐     ┌─────────────┐
│  判断任务类型  │ ──▶ │ feature-dev 或    │ ──▶ │ code-review │
│              │     │ bug-fix 完整流程   │     │ (自动衔接)   │
└─────────────┘     └──────────────────┘     └─────────────┘
```

---

## 阶段一：执行开发/修复流程

### 步骤 1：识别任务类型

根据用户请求判断属于哪种类型：

- **功能开发类**（"开发"、"实现"、"新增"） → 执行 `feature-dev` Skill 的完整流程
- **Bug 修复类**（"修复"、"bug"、"排查"、"异常"） → 执行 `bug-fix` Skill 的完整流程

### 步骤 2：完整执行对应 Skill

严格按照对应 Skill 定义的全部阶段执行，**不得跳过任何步骤**，包括：
- 前置需求澄清（`requirement-clarify`）
- 方案设计 / 问题定位（需等待用户确认）
- 编码实现 / 修复实施
- 自检清单
- 变更摘要
- ChangeLog 输出

### 步骤 3：记录变更信息

在开发/修复流程的**自检与交付阶段完成后**，收集以下信息供评审阶段使用：
- **变更文件清单**：所有新增和修改的文件路径
- **业务背景**：本次变更实现的功能或修复的问题
- **变更类型**：新功能 / Bug 修复

---

## 阶段二：自动衔接代码评审

> ⚠️ 开发/修复流程全部完成后，**自动进入评审阶段，无需用户再次发起请求**。

### 步骤 4：过渡提示

输出过渡说明，让用户知道即将进入评审：

```markdown
---

## 🔍 进入代码评审阶段

开发/修复工作已完成，现在自动进入代码评审流程。

- **审查范围**: [列出本次变更的所有文件]
- **业务背景**: [一句话描述本次变更的目的]
- **变更类型**: [新功能 / Bug 修复]
```

### 步骤 5：跳过评审需求澄清

由于审查范围和业务背景已在开发/修复阶段完全确定，**跳过 `requirement-clarify` 中 code-review 的追问环节**，直接将以下信息视为已确认：

| code-review 必问项 | 来源 |
|:---:|------|
| R-01 审查范围 | 阶段一记录的变更文件清单 |
| R-02 代码业务背景 | 阶段一的需求描述和方案设计 |

选问项自动填充：
- R-03 重点关注维度 → **全维度审查**
- R-04 变更上下文 → 阶段一的任务类型（新功能 / Bug 修复）

### 步骤 6：执行三维度审查

严格按照 `code-review` Skill 的**步骤 2（三维度系统审查）** 执行：

1. **阅读 `AGENTS.md` 安全规范**作为审查基线
2. 按安全审查（🔴）→ 规范审查（🟡）→ 质量审查（🟢）顺序逐项检查
3. 每个检查项给出 ✅ 或 ❌ 判定
4. **审查范围限定为本次变更的文件**，不审查未变更的代码

### 步骤 7：输出审查报告

按照 `code-review` Skill 的**步骤 3** 格式输出完整审查报告。

### 步骤 8：问题处理

根据审查结果分支处理：

**情况 A — 存在 🔴 阻断问题**：
1. 明确告知用户存在阻断问题，建议修复后再合入
2. 询问用户是否需要协助修复
3. 如果用户同意，按严重等级从高到低依次修复
4. 修复后重新执行步骤 6-7，确认问题已清零

**情况 B — 仅有 🟡/🟢 问题**：
1. 输出审查报告
2. 告知用户可安全合入，建议在本迭代内处理 🟡 问题

**情况 C — 无问题**：
1. 输出"审查通过"报告
2. 确认代码质量良好，可安全合入

---

## 最终交付物

流程完成后，用户应获得以下完整交付：

1. ✅ 开发/修复的完整代码变更
2. ✅ 变更摘要（来自 feature-dev / bug-fix）
3. ✅ ChangeLog 条目（来自 feature-dev / bug-fix）
4. ✅ 代码评审报告（来自 code-review）
5. ✅ 阻断问题已修复（如有）
6. ✅ 任务交付文档（输出至 `docs/` 目录）

---

## 阶段三：生成任务交付文档

> ⚠️ **必须在所有步骤完成后执行**，使用 Write 工具将文档写入 `docs/` 目录。
> 
> **注意**：dev-review 流程生成**一份合并文档**，涵盖开发/修复和评审两个阶段的全部信息，**不再**由子 Skill（feature-dev / bug-fix / code-review）各自单独生成文档。

文件命名规则：`docs/{YYYY-MM-DD}-dev-review-{功能或问题简称}.md`（简称用英文小写短横线分隔，如 `user-register`、`fix-login-npe`）。

文档模板如下（按实际情况填写，无变更的章节保留标题并标注"无"）：

```markdown
# 任务交付文档

## 基本信息
- **任务类型**: [功能开发 + 代码评审 / Bug 修复 + 代码评审]
- **执行日期**: YYYY-MM-DD
- **执行 Skill**: dev-review（feature-dev/bug-fix + code-review）

---

## 一、任务描述
[本次任务的完整背景、业务目标或问题描述]

## 二、实现/修复方案

### 2.1 方案概述
[技术方案或修复思路的核心内容]

### 2.2 接口设计（功能开发时填写）
| 路径 | 方法 | 功能 | 入参 | 出参 |
|------|------|------|------|------|

### 2.3 根因分析（Bug 修复时填写）
- **根因位置**: [文件名:行号]
- **问题类型**: [逻辑错误 / 数据问题 / 配置问题 / 并发问题 / 依赖问题]
- **原因说明**: [详细解释]

## 三、变更清单

### 3.1 文件变更
| 文件路径 | 变更类型 | 变更说明 |
|----------|:---:|----------|

### 3.2 配置变更
| 配置项/环境变量 | 变更类型 | 值/默认值 | 说明 | 是否敏感 |
|----------------|:---:|----------|------|:---:|

### 3.3 数据库变更
| 变更类型 | 表/对象 | 说明 |
|:---:|----------|------|

### 3.4 依赖变更
| GroupId | ArtifactId | 版本 | 变更类型 | 用途 |
|---------|------------|------|:---:|------|

## 四、开发自检结果
[粘贴 feature-dev / bug-fix 自检清单的通过状态摘要]

## 五、代码评审结论

### 5.1 问题汇总
| 序号 | 文件:行号 | 严重等级 | 检查项 | 问题描述 | 修复建议 |
|:---:|----------|:---:|:---:|----------|----------|

### 5.2 统计
- 🔴 阻断问题: X 个（已修复 / 待修复）
- 🟡 警告问题: X 个
- 🟢 建议问题: X 个

### 5.3 总体评价
[综合评价代码质量，指出主要亮点和薄弱环节]

## 六、风险评估
- **回归风险**: [低/中/高 + 理由]
- **安全风险**: [是否存在未修复的安全问题]
- **影响范围**: [受影响的模块/服务]

## 七、ChangeLog 条目
[粘贴 feature-dev / bug-fix 生成的 ChangeLog 条目]

## 八、后续建议
- [需要后续跟进的事项，如补充测试、性能优化、监控配置等]
```
