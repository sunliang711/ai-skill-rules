# Go 功能开发 SOP

> 当用户提出“开发 Go 功能”、“实现 Gin 接口”、“新增后台任务”、“补一个 Go 服务能力”等请求时使用。

## 前置：需求澄清

> 必须先执行 `requirement-clarify-go`，至少确认 `FG-01` ~ `FG-04` 后再继续。

## 阶段一：需求理解与方案设计

### 步骤 1：整理已知信息
确认以下内容已经明确：
- 业务目标与边界
- 影响的包和分层
- 接口形态与调用方式
- 核心规则、异常分支、并发与事务要求

- 用户确认方案后再编码。

## 阶段二：编码实现

> 编码时必须遵循 `rules/04-go-backend.mdc`；涉及鉴权、密钥、日志脱敏等安全能力时，还必须遵循 `rules/05-go-security.mdc`；涉及 Gin Handler / HTTP 接口时，再叠加 `rules/06-go-api-design.mdc`。

### 步骤 4：按推荐顺序实现

#### 4.1 配置与常量
- 新增配置先补配置结构、默认值、校验逻辑。
- 禁止在业务代码中散落调用 `viper.Get*()`。

#### 4.2 Model / DTO
- 数据库模型与对外 DTO 分离。
- JSON、GORM、`mapstructure`、`binding`/`validate` tag 顺序保持一致。

#### 4.3 Repository
- Repository 只负责数据访问。
- 所有数据库查询必须带 `ctx`。
- 事务优先通过 `db.Transaction` 控制。

#### 4.4 Service
- 业务逻辑放在 Service。
- 通过构造函数注入依赖，避免全局状态。
- 涉及并发时显式处理共享状态、goroutine 生命周期和取消信号。

#### 4.5 Handler / 入口层
- Handler 只做参数绑定、校验、响应封装。
- Gin 路由按模块注册，路径遵循 `/api/v{n}/{resource}` 规范。
- 错误统一转换为标准响应，禁止在下层直接写 HTTP 响应。

#### 4.6 启动与注入
- 依赖通过 Fx 构造函数注入。
- 资源初始化和关闭走 `fx.Lifecycle`。

### 步骤 5：注释与可读性
- 仅为复杂业务规则、并发控制、非直观取舍补简洁注释。
- 不添加解释显而易见代码的噪声注释。

## 阶段三：测试、自检与交付

### 步骤 6：测试要求
- 为新增或改动的核心逻辑补 Go 测试。
- 优先表驱动测试。
- 对外部依赖使用 mock 或 stub。
- Handler 测试优先使用 `httptest`。
- 需要数据库或中间件时，优先隔离式集成测试，不在单元测试中连真实外部服务。

### 步骤 7：自检清单

#### 架构
- [ ] Handler 无业务逻辑
- [ ] Service 无 HTTP 细节
- [ ] Repository 无业务判断
- [ ] DTO 与 Model 分离

#### Go 规范
- [ ] `context.Context` 传递正确
- [ ] `err` 都被处理或包装
- [ ] 无无意义全局变量和 `init()` 副作用
- [ ] 无魔法数字和字符串拼接日志

#### 稳定性与安全
- [ ] 并发共享状态已保护
- [ ] goroutine 有退出机制
- [ ] 日志无敏感信息
- [ ] 配置与凭据未硬编码
- [ ] 金额使用 `decimal.Decimal`（如适用）

#### 测试
- [ ] 正常路径已覆盖
- [ ] 错误路径已覆盖
- [ ] 边界条件已覆盖
- [ ] 相关测试可运行
