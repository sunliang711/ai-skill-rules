# Python 重构 SOP

> 当用户提出“重构 Python 代码”、“优化 FastAPI / Service 结构”、“拆分类和函数”等请求时使用。

## 前置：需求澄清

> 必须先执行 `requirement-clarify-python`，至少确认 `RFP-01` ~ `RFP-03`。

## 阶段一：确认范围与约束

### 步骤 1：明确重构目标
- 为什么重构
- 具体涉及哪些模块、类、函数、任务或路由
- 哪些外部行为、配置键、脚本参数或 API 契约必须保持不变

### 步骤 2：先输出重构方案

```markdown
## 重构方案
- 重构目标：
- 影响范围：
- 保持不变的行为：
- 分步实施计划：
- 回归验证方式：
```

## 阶段二：小步实施

> 编码时遵循 `rules/10-python-backend.mdc`；涉及安全和 HTTP 时叠加 `rules/11-python-security.mdc`、`rules/12-python-api-design.mdc`。

### 步骤 3：按小步推进
- 一次只做一类结构调整
- 先提取函数，再整理模块，再调整依赖方向
- 禁止在重构过程中顺带改动对外接口和业务语义
- 异步任务、线程、缓存和会话边界变化必须单独评估

## 阶段三：回归验证与交付

### 步骤 4：验证重点
- 相关测试和静态检查是否通过
- 错误处理、日志字段、配置读取和事务语义是否保持一致
- 是否引入新的循环依赖、全局状态或资源泄漏

### 步骤 5：输出结果

```markdown
## 重构摘要
- 目标：
- 改动模块：
- 保持不变的行为：
- 验证结果：
- 残余风险：
```
