# Python 功能开发 SOP

> 当用户提出“开发 Python 功能”、“实现 FastAPI 接口”、“补一个 Python 任务/脚本”等请求时使用。

## 前置：需求澄清

> 必须先执行 `requirement-clarify-python`，至少确认 `FP-01` ~ `FP-04`。

## 阶段一：理解需求与方案设计

### 步骤 1：整理已知信息
- 业务目标、边界和关键规则
- 影响的包、模块、服务、任务或脚本
- 数据、外部依赖、权限和异常场景

### 步骤 2：先输出实现方案

```markdown
## 实现方案
- 影响范围：
- 入口类型：HTTP / Worker / CLI / Script / Library
- 核心流程：
- 数据与依赖变更：
- 配置项变更：
- 风险与验证方式：
```

### 步骤 3：等待确认
- 用户确认后再编码
- 需求变化先更新方案再继续

## 阶段二：编码实现

> 编码时必须遵循 `rules/10-python-backend.mdc`；涉及密钥、鉴权、注入、命令执行或 TLS 时，还必须遵循 `rules/11-python-security.mdc`；涉及 HTTP API 时，再叠加 `rules/12-python-api-design.mdc`。

### 步骤 4：按推荐顺序实现
- 先补配置、Schema / DTO、常量和校验
- Repository / DAO 只负责数据访问，Service 负责业务逻辑
- Router / View / CLI 入口只做参数接收、校验、响应封装
- 新代码优先补类型注解，避免大面积裸字典和隐式结构
- 禁止 `except Exception: pass`、禁止无说明吞错
- 异步任务、线程、后台 worker 必须明确超时、取消和关闭逻辑

## 阶段三：测试、自检与交付

### 步骤 5：测试要求
- 为新增或改动的核心逻辑补测试
- 覆盖正常路径、错误路径、边界条件以及权限/超时场景
- Web 层优先使用测试客户端，外部依赖优先 mock 或 fake
- 至少说明应执行的检查，如 `pytest`、`ruff`、`mypy`

### 步骤 6：自检清单
- [ ] 分层清晰，入口层无业务逻辑
- [ ] 输入校验和错误处理完整
- [ ] 配置、凭据、路径未硬编码
- [ ] 日志无敏感信息
- [ ] 事务、分页、超时与资源释放合理
- [ ] 测试覆盖核心路径并可运行

### 步骤 7：输出交付摘要

```markdown
## 变更摘要
- 新增文件：
- 修改文件：
- 新增接口 / 任务 / 命令：
- 配置或依赖变更：
- 测试情况：
```

### 步骤 8：任务交付文档
在 `docs/` 下生成：`docs/{YYYY-MM-DD}-feature-python-{功能简称}.md`
