# Go 开发 + 评审组合流程

> 当用户的请求同时包含“开发/修复”和“做完后 review”两层意图时使用。

## 流程概览

```text
判断任务类型
-> feature-dev-go 或 bug-fix-go
-> code-review-go
-> 合并交付
```

## 阶段一：先完成开发或修复

### 步骤 1：识别任务类型
- 新增功能、接口、任务能力：执行 `feature-dev-go`
- 修复错误、panic、异常行为：执行 `bug-fix-go`

### 步骤 2：完整执行对应 Skill
不得跳过以下环节：
- `requirement-clarify-go`
- 方案设计或定位结论
- 编码实现
- Go 测试
- 自检与变更摘要

### 步骤 3：收集评审输入
进入评审前整理：
- 本次新增和修改的文件
- 业务背景
- 变更类型
- 测试结果

## 阶段二：自动衔接评审

### 步骤 4：跳过重复澄清
开发或修复阶段已经明确的内容，可直接作为 `code-review-go` 的输入：
- 审查范围：本次变更文件
- 业务背景：阶段一已确认内容
- 变更类型：新功能或 Bug 修复

### 步骤 5：执行 Go 代码评审
- 按 `code-review-go` 的安全、规范、质量三维度审查。
- 评审范围只限本次变更文件。

### 步骤 6：根据审查结果处理
- 有阻断问题：优先修复，再复审。
- 只有警告或建议：输出结果并说明风险。
- 无问题：输出通过结论和残余风险。

## 阶段三：最终交付

用户最终应获得：
- 已完成的 Go 代码变更
- 测试结果
- 变更摘要
- 代码评审报告
- 风险结论

## 任务交付文档
在 `docs/delivery/` 下生成：`docs/delivery/{YYYY-MM-DD-HH-mm-ss}-dev-review-go-{功能或问题简称}.md`

建议包含：
- 任务背景
- 实现或修复方案
- 文件与配置变更
- 测试结果
- 评审问题清单
- 风险与后续建议
