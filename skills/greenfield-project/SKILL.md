# 从零构建项目 SOP（Greenfield Project）

> 当用户提出"从零开始做一个项目"、"搭建新系统"、"新建项目"等从头构建类请求时，严格按以下流程执行。

---

## 核心原则

1. **理解先于方案** — 禁止在需求不清晰时输出架构或编码
2. **方案先于编码** — 禁止在方案未经用户确认时动手实现
3. **渐进式深入** — 每轮追问 ≤ 5 个问题，由粗到细
4. **文档即契约** — 所有确认的决策都落入文档，后续开发以文档为准
5. **可恢复** — 通过进度文档，中断后可无缝衔接

---

## 阶段一：需求发现

### 步骤 1：初始信息收集

收到用户请求后，围绕以下维度进行结构化追问。每轮 ≤ 5 个问题。

#### 🔴 必问项

| 编号 | 维度 | 核心问题 |
|:---:|------|---------|
| G-01 | **项目愿景** | 这个项目要解决什么问题？服务谁？核心价值主张是什么？ |
| G-02 | **核心功能** | 列出必须实现的核心功能（MVP 范围）。哪些是第一期必须交付的？ |
| G-03 | **技术栈偏好** | 前端/后端/数据库/基础设施有没有指定的技术栈？有没有必须用或不能用的技术？ |
| G-04 | **用户角色** | 系统有哪些角色？各角色的核心操作是什么？ |
| G-05 | **部署环境** | 部署在哪里？（云/本地/混合）有没有环境约束？ |

#### 🟡 选问项（根据项目类型选择性追问）

| 编号 | 维度 | 问题 |
|:---:|------|------|
| G-06 | **非功能性需求** | 预期并发量、响应时间、可用性要求？ |
| G-07 | **集成需求** | 需要对接哪些外部系统/API/第三方服务？ |
| G-08 | **数据需求** | 核心数据实体有哪些？数据量级和增长预估？ |
| G-09 | **安全需求** | 认证方式？数据加密要求？合规要求？ |
| G-10 | **团队背景** | 开发团队的人数和技术背景？（影响技术选型复杂度） |
| G-11 | **时间约束** | 有没有里程碑节点或截止日期？ |
| G-12 | **已有资产** | 有没有现成的设计稿、API 文档、数据库 schema、竞品参考？ |
| G-13 | **扩展规划** | 未来可能追加的功能方向？需要预留哪些扩展点？ |

### 步骤 2：循环澄清

- 用户回答后，更新内部认知，识别新的模糊点
- 如仍有关键未知项，进行下一轮追问（最多 4 轮）
- 可以合理推断并说明："我理解为 XXX，如有偏差请纠正"
- 4 轮后仍有模糊点，标注为"待定"并在方案中高亮

### 步骤 3：需求确认

所有必问项确认后，输出一段**需求理解摘要**，用自己的话复述：
- 项目定位与核心价值
- 核心功能清单（按优先级排列）
- 技术栈初步选择
- 关键约束与边界

等待用户确认："以上理解是否正确？"

---

## 阶段二：方案设计

### 步骤 4：输出整体方案

需求确认后，输出完整的架构方案文档（使用 [templates.md](templates.md) 中的「整体方案模板」）。方案必须包含：

1. **项目概述** — 一段话概括项目
2. **业务需求** — 功能清单，按模块分组，标注优先级
3. **系统架构** — 整体架构描述（单体/微服务/Serverless），含架构图（文字描述或 Mermaid）
4. **技术栈决策** — 每项技术选择附带选型理由
5. **模块划分** — 系统分成哪些模块/服务，各自职责
6. **数据模型** — 核心实体及其关系
7. **接口设计概览** — 主要 API 端点分组列出
8. **非功能性设计** — 性能、安全、可观测性方案
9. **部署方案** — 环境规划、CI/CD 流程
10. **任务分解** — 拆分为有序子任务（含依赖关系）
11. **风险登记** — 已识别的技术风险与应对策略
12. **领域术语表** — 项目中使用的专用术语统一定义
13. **架构决策记录（ADR）** — 关键技术决策的背景、决定和后果

---

## 阶段三：方案评审与迭代

### 步骤 5：方案辩论

- 向用户呈现方案，主动指出需要用户重点关注的决策点
- 逐一回应用户的质疑和修改意见
- 每次修改后输出变更的部分（增量更新，不重复输出未变更内容）
- **禁止在用户明确表示"确认"/"通过"/"同意"前进入下一阶段**

### 步骤 6：用户确认

用户确认方案后：
1. 询问用户期望的项目文档输出目录（默认：`docs/project/{project-name}/`）
2. 确认子任务的粒度是否合适

---

## 阶段四：文档输出

### 步骤 7：生成整体方案文档

使用 Write 工具，按照 [templates.md](templates.md) 中的「整体方案模板」生成：

```
docs/project/{project-name}/architecture.md
```

### 步骤 8：生成子任务文档

为每个子任务生成独立文档，按照 [templates.md](templates.md) 中的「子任务模板」：

```
docs/project/{project-name}/tasks/task-{nn}-{short-name}.md
```

每个子任务文档包含：
- 任务描述与目标
- 技术方案
- 验收标准（Definition of Done）
- 实现步骤指引
- 前置依赖与风险

### 步骤 9：生成进度文档

按照 [templates.md](templates.md) 中的「进度跟踪模板」生成：

```
docs/project/{project-name}/PROGRESS.md
```

### 步骤 10：生成编码规范文档（推荐）

根据选定的技术栈，输出项目专属的编码规范：

```
docs/project/{project-name}/conventions.md
```

内容包括：目录结构规范、命名规范、错误处理规范、日志规范、Git 提交规范等。

### 步骤 11：输出文档清单

所有文档生成后，输出文件树和简要说明，方便用户总览：

```markdown
📁 docs/project/{project-name}/
├── architecture.md        # 整体架构方案
├── conventions.md         # 编码规范
├── PROGRESS.md            # 进度跟踪
└── tasks/
    ├── task-01-xxx.md     # 子任务 1
    ├── task-02-xxx.md     # 子任务 2
    └── ...
```

---

## 阶段五：进入开发（衔接）

文档输出完成后，提示用户：

> 📋 项目规划已完成。你可以：
> 1. 选择一个子任务开始开发（告诉我任务编号）
> 2. 调整任务优先级或拆分粒度
> 3. 先 review 某个子任务的方案

当用户选择开发某个子任务时：
1. 读取对应的子任务文档
2. 读取进度文档了解当前状态
3. 根据技术栈触发对应的开发 Skill（如 `feature-dev-java`、`feature-dev-go`）
4. 完成后更新进度文档

## 特殊场景处理

### 用户说"别问了直接出方案"
- 尊重用户意愿，基于已有信息直接进入阶段二
- 在方案中标注所有基于假设的决策（⚠️ 假设标记）

### 已有部分设计资料
- 先阅读用户提供的资料（设计稿、API 文档等）
- 减少重复追问，聚焦缺失的信息

### 恢复中断的项目
- 如果 `PROGRESS.md` 已存在，先读取进度和方案文档
- 向用户确认是否继续上次的进度
- 从断点处衔接

### 项目类型特化
根据不同项目类型，追问侧重点不同：
- **Web 应用**: 重点关注前后端分离方式、状态管理、SEO
- **API 服务**: 重点关注认证鉴权、限流、文档生成
- **CLI 工具**: 重点关注参数设计、交互方式、跨平台
- **移动应用**: 重点关注平台（iOS/Android/跨平台）、离线策略
- **数据处理**: 重点关注数据源、处理量级、实时性要求
