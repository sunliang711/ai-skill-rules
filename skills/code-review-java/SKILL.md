# 代码评审 SOP（Standard Operating Procedure）

> 当用户提出"审查代码"、"review"、"帮我看看这段代码"等评审类请求时，严格按以下流程执行。

---

## 前置：需求澄清（强制）

> ⚠️ **必须先执行 `requirement-clarify-java` Skill 中的澄清协议，通过后方可进入正式审查流程。**
> 
> 对照 `requirement-clarify-java` 中「code-review 专属问题清单」的 🔴 必问项（R-01 ~ R-02）进行信息充分度自检，
> 缺失项必须向用户追问，**禁止跳过此环节直接开始审查**。

---

## 执行流程

### 步骤 1：准备审查
> 此步骤承接「前置：需求澄清」的输出。审查范围和业务背景已在 `requirement-clarify-java` 中确认。

1. 根据已确认的审查范围，列出待审文件清单
2. 如果涉及整个目录/模块，先列出文件清单供用户确认后再逐文件审查
3. 阅读并理解待审代码的业务上下文和变更意图

### 步骤 2：三维度系统审查
> ⚠️ **审查时必须以 `.cursor/rules/02-java-security.mdc` 中的安全红线规范为审查基线。**

按以下三个维度**逐项检查**，每个检查项必须给出明确的 ✅（通过）或 ❌（不通过）判定：

---

### 维度一：安全审查（优先级最高 🔴）

| 编号 | 检查项 | 说明 |
|:---:|--------|------|
| S-01 | 硬编码凭据 | 代码中是否存在硬编码的密码、Key、Token、连接串 |
| S-02 | SQL 注入 | 是否全部使用参数化查询，有无字符串拼接 SQL |
| S-03 | 日志泄密 | 日志是否输出了私钥、密码、PIN、Token、完整签名数据等 |
| S-04 | 异常暴露 | 异常响应是否泄露了堆栈、SQL、框架内部信息 |
| S-05 | 认证授权 | 接口是否有认证拦截和权限控制 |
| S-06 | 输入校验 | 外部输入是否经过校验（@Valid、自定义校验） |
| S-07 | SSRF/XXE | 是否存在 URL 注入、XML 外部实体等风险 |
| S-08 | 反序列化 | 是否使用了不安全的反序列化方式 |
| S-09 | TLS 配置 | 是否存在信任所有证书、关闭主机名校验等配置 |
| S-10 | 敏感数据存储 | 密码是否用 BCrypt/Argon2 哈希，是否有明文存储 |
| S-11 | 密钥/私钥内存清除 | 敏感 byte[]/char[] 使用后是否 Arrays.fill() 清零；是否用 String 传递密钥/PIN |
| S-12 | 金额精度 | 金额/余额/手续费是否使用 BigDecimal；是否误用 double/float |
| S-13 | 安全开关可覆盖 | SSL、Vault 等安全开关是否可被环境变量 `${XX:false}` 关闭 |
| S-14 | 签名/交易防重放 | 签名请求是否有 nonce/timestamp 防重放机制 |

---

### 维度二：规范审查 🟡

| 编号 | 检查项 | 说明 |
|:---:|--------|------|
| N-01 | 分层架构 | Controller 是否包含业务逻辑；Service 是否直接访问数据库 |
| N-02 | 命名规范 | 类名、方法名、变量名是否符合团队约定 |
| N-03 | DTO 分离 | Entity 是否直接暴露给了 Controller 层 |
| N-04 | REST 路径 | API 路径是否符合规范（小写、连字符、复数、版本号） |
| N-05 | 响应格式 | 是否使用了统一的响应封装结构 |
| N-06 | HTTP 语义 | GET 是否用于查询、POST 用于创建（有无语义错用） |
| N-07 | 常量管理 | 是否存在魔法数字、硬编码字符串 |
| N-08 | 注释完整性 | 关键业务逻辑是否有注释说明 |

---

### 维度三：质量审查 🟢

| 编号 | 检查项 | 说明 |
|:---:|--------|------|
| Q-01 | 空指针风险 | 方法返回值、集合元素是否做了空值检查 |
| Q-02 | 资源泄漏 | 数据库连接、I/O 流是否使用 try-with-resources |
| Q-03 | 并发安全 | 共享状态是否加锁、非线程安全类是否跨线程共享 |
| Q-04 | 边界检查 | 集合操作、数组访问是否有越界保护 |
| Q-05 | 分页上限 | 分页查询是否设置了 MAX_PAGE_SIZE |
| Q-06 | 事务管理 | @Transactional 使用是否正确（传播级别、回滚规则） |
| Q-07 | 异常处理 | 是否有空 catch 块、是否吞掉了异常 |
| Q-08 | 重复代码 | 是否有可提取为公共方法的重复逻辑 |
| Q-09 | 性能隐患 | 是否有循环中的数据库查询（N+1）、不必要的全表扫描 |
| Q-10 | 代码简洁性 | 是否有过度嵌套（层数 > 3）、过长方法（行数 > 80）|

---

### 步骤 3：输出审查报告

以表格形式输出所有发现的问题，格式如下：

```markdown
## 代码评审报告

### 审查文件
- [文件路径列表]

### 发现的问题

| 序号 | 文件:行号 | 严重等级 | 检查项 | 问题描述 | 修复建议 |
|:---:|----------|:---:|:---:|----------|----------|
| 1 | UserService.java:45 | 🔴 阻断 | S-03 | 日志输出了用户手机号明文 | 使用 MaskUtil.maskPhone() 脱敏 |
| 2 | OrderController.java:22 | 🟡 警告 | N-01 | Controller 中包含了订单金额计算逻辑 | 移至 OrderServiceImpl |
| 3 | UserMapper.xml:18 | 🔴 阻断 | S-02 | 使用 ${username} 拼接 SQL | 改为 #{username} 参数化查询 |

### 审查总结
- 🔴 阻断问题: X 个（必须修复后方可合入）
- 🟡 警告问题: X 个（建议在本次迭代修复）
- 🟢 建议问题: X 个（可后续优化）

### 总体评价
[综合评价代码质量，指出主要亮点和薄弱环节]
```

### 步骤 4：修复确认（可选）
如果用户要求协助修复：
1. 按照严重等级从高到低依次修复
2. 每次修复后说明改了什么、为什么这样改
3. 修复完成后重新输出审查报告，确认所有问题已清零

### 特殊情况：审查未发现问题
如果全部检查项均通过，输出：

```markdown
## 代码评审报告

### 审查文件
- [文件路径列表]

### 审查结果
✅ 本次审查未发现问题，代码质量良好。

### 亮点
- [列出代码中值得肯定的做法]
```
