# Rust 重构 SOP

> 当用户提出“重构 Rust 代码”、“优化 Rust 模块结构”、“拆 trait / impl / async 逻辑”等请求时使用。

## 前置：需求澄清

> 必须先执行 `requirement-clarify-rust`，至少确认 `RFR-01` ~ `RFR-03`。

## 阶段一：确认范围与约束

### 步骤 1：明确重构目标
- 为什么重构：降复杂度、解耦、提取共用逻辑、提升测试性或优化边界
- 具体涉及哪些模块、trait、impl、handler、service
- 哪些外部行为、协议、命令行接口或配置语义必须保持不变

### 步骤 2：先输出重构方案

```markdown
## 重构方案
- 重构目标：
- 影响范围：
- 保持不变的外部行为：
- 分步实施计划：
- 回归验证方式：
```

## 阶段二：小步实施

> 编码时遵循 `rules/07-rust-backend.mdc`；涉及安全和 HTTP 时叠加 `rules/08-rust-security.mdc`、`rules/09-rust-api-design.mdc`。

### 步骤 3：按小步推进
- 优先提取边界清晰的函数、模块或 trait
- 每步只做一种结构调整，避免语义变化与结构变化交织
- 不随意更改公开接口、错误码、配置键和数据格式
- 异步边界、锁粒度、共享状态变化必须单独评估

## 阶段三：回归验证与交付

### 步骤 4：验证重点
- 相关测试、编译和静态检查是否通过
- 错误链、日志字段、配置读取是否仍然一致
- 并发和生命周期是否引入新风险

### 步骤 5：输出结果

```markdown
## 重构摘要
- 目标：
- 改动模块：
- 保持不变的行为：
- 验证结果：
- 残余风险：
```
