# Java API 文档 SOP

> 当用户提出“生成 Java API 文档”、“整理接口说明”、“补 curl 调用示例”、“输出 Swagger 对应文档说明”等请求时使用。

## 前置：需求澄清

> 必须先执行 `requirement-clarify-java`，至少确认 `A-01` ~ `A-03`。

## 步骤 1：确认文档范围
- 涉及哪些 Controller、接口分组、模块或服务
- 文档面向对内调用、对外开放还是测试联调
- 输出目标是 Markdown、Swagger / SpringDoc 注解补充，还是两者都需要

优先扫描：
- `controller/`、`web/`
- 请求 / 响应 DTO
- 鉴权注解、拦截器、网关约束
- 全局异常处理、错误码定义
- 现有 Swagger / SpringDoc 注解、README、联调文档

## 步骤 2：提取接口契约

### 2.1 基本信息
- 接口分组、方法、路径、版本号
- 接口用途、调用方、是否幂等

### 2.2 请求信息
- Path / Query / Header / Cookie / Body 参数
- 字段类型、是否必填、默认值、取值范围、枚举值
- 鉴权头、签名、时间戳、分页等通用要求

### 2.3 响应信息
- 成功响应结构
- 失败响应结构
- HTTP 状态码与业务错误码映射
- 脱敏字段、追踪字段、分页字段

### 2.4 示例信息
- `curl` 调用示例
- 请求体示例
- 成功响应示例
- 常见错误响应示例

> 示例数据必须脱敏；如缺少真实样例，基于代码结构构造最小可理解示例，并标注为示例值。

## 步骤 4：同步现有文档机制
- 若项目已启用 Swagger / SpringDoc，补齐 `@Operation`、`@Parameter`、`@Schema` 等注解
- 若已有接口文档目录，沿用既有命名和组织方式
- 若用户要求 OpenAPI 产物，优先基于现有文档机制同步，而不是手写与代码脱节的 spec

## 步骤 5：交叉验证
- 路径、方法、鉴权要求与 Controller 实现一致
- DTO 字段名、必填约束、枚举值与代码一致
- `curl` 示例中的 Header、方法、路径和 Body 可直接映射到实现
- 错误码、异常响应与全局异常处理器一致
- 示例值已脱敏，未泄露真实密钥、Token、手机号等敏感信息

## 步骤 6：任务交付文档
在 `docs/api/` 下生成：`docs/api/{YYYY-MM-DD}-api-doc-java-{模块简称}.md`

建议包含：
- 文档范围
- 提取来源
- API 文档正文
- 与 Swagger / SpringDoc 的同步情况
- 风险与待确认项
