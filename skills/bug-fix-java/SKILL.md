# Bug 修复 SOP（Standard Operating Procedure）

> 当用户提出"修复XXX问题"、"这里有个bug"、"排查XXX异常"等 Bug 修复类请求时，严格按以下流程执行。

---

## 前置：需求澄清（强制）

> ⚠️ **必须先执行 `requirement-clarify-java` Skill 中的澄清协议，通过后方可进入阶段一。**
> 
> 对照 `requirement-clarify-java` 中「bug-fix 专属问题清单」的 🔴 必问项（B-01 ~ B-03）进行信息充分度自检，
> 缺失项必须向用户追问，**禁止跳过此环节直接猜测原因或修改代码**。

---

## 阶段一：问题理解与定位

### 步骤 1：信息整理
> 此步骤承接「前置：需求澄清」的输出。追问环节已在 `requirement-clarify-java` 中完成。

1. 梳理需求澄清阶段已确认的信息，确保以下要素就绪：
   - 问题现象（报错信息、异常行为）
   - 复现条件（操作路径、数据条件、环境）
   - 期望行为（正确结果是什么）
   - 影响范围与紧急程度
2. 如在定位过程中发现新的模糊点，可简要追问后继续

### 步骤 2：问题定位
1. 阅读相关代码，理解业务逻辑和数据流
2. 从异常信息/日志出发，追踪调用链定位根因
3. 区分是以下哪类问题：
   - **逻辑错误**: 条件判断、状态流转、边界处理
   - **数据问题**: 空值、类型转换、精度丢失
   - **配置问题**: 环境变量缺失、配置项错误
   - **并发问题**: 竞态条件、死锁、线程安全
   - **依赖问题**: 版本冲突、接口变更
4. 输出定位结论，**等待用户确认后再进入修复阶段**：

```markdown
## Bug 定位分析

### 问题现象
[复述用户描述的问题现象]

### 根因分析
- 根因位置: [文件名:行号]
- 问题类型: [逻辑错误 / 数据问题 / 配置问题 / 并发问题 / 依赖问题]
- 原因说明: [详细解释为什么会出现这个问题]
- 触发条件: [什么情况下会触发]

### 修复方案
- 修改文件: [列出需要修改的文件]
- 修改思路: [说明怎么改、为什么这样改]
- 影响评估: [修改是否可能影响其他功能]

### 是否需要同步修改
- [ ] 是否有其他模块存在相同问题（如 Online/Offline 同步）
- [ ] 是否需要同步更新测试用例
```

---

## 阶段二：修复实施

> ⚠️ **修复编码前必须遵循 `.cursor/rules/02-java-security.mdc` 中的安全红线规范**，确保修复代码符合团队安全基线。
> 
> 🔴 **核心原则：全盘考量，不引入新问题。** 修复 Bug 时必须充分理解上下游调用链和关联模块，评估修改对整体系统的影响。任何修复都不能以引入新的 Bug、破坏已有功能或降低系统稳定性为代价。

### 步骤 3：编码修复
1. **只修改与 Bug 直接相关的代码**，严禁"顺手"重构
2. 遵循现有代码风格，最小化改动
3. **全盘影响评估**（强制）：
   - 修改前梳理该方法/类的所有调用方，确认修复不会破坏已有调用者的预期行为
   - 检查修改是否涉及共享数据结构、公共工具方法或全局配置，评估波及范围
   - 如修改涉及接口契约（入参/出参/异常类型）变更，必须同步排查并更新所有调用方
   - 并发场景下的修复必须考虑线程安全，避免引入竞态条件
4. 修复代码必须添加注释说明：
   - 修复了什么问题
   - 为什么这样修而不是那样修（如有多种方案时）

### 步骤 4：防御性加固（如适用）
如果 Bug 暴露了代码中的系统性薄弱环节，在修复的同时进行必要的防御加固：
- 入参增加空值/边界检查
- 添加必要的异常捕获
- 补充日志输出便于未来排查（注意脱敏）

---

## 阶段三：验证与交付

### 步骤 5：自检清单
修复完成后，逐条对照检查：

#### 修复有效性
- [ ] 修复后原问题不再复现
- [ ] 修复逻辑覆盖了所有已知触发条件
- [ ] 没有引入新的问题（回归风险）
- [ ] 已梳理所有调用方，确认修复不破坏已有功能
- [ ] 共享数据结构/公共方法的修改已评估全部影响范围

#### 代码规范
- [ ] 只改了必要的代码，无多余修改
- [ ] 代码风格与周围一致
- [ ] 无安全隐患（日志脱敏、异常处理等）

#### 同步检查
- [ ] 相同代码在其他模块是否存在同样问题（是否需要同步修复）
- [ ] 相关单元测试是否需要更新

### 步骤 6：变更摘要
输出简洁的修复摘要：

```markdown
## Bug 修复摘要

### 问题
- 问题描述: [一句话描述]
- 影响范围: [受影响的功能/接口]

### 修复
- 修改文件: [列出修改的文件和行号]
- 修复方式: [一句话说明怎么修的]
- 根因: [一句话说明为什么出问题]

### 验证
- 复现测试: [修复后是否已验证不再复现]
- 回归风险: [低/中/高 + 理由]
```

### 步骤 7：ChangeLog 输出
输出一条标准格式的 ChangeLog 条目：

```markdown
## [版本号] - YYYY-MM-DD

### Fixed
- [功能名称]: 修复 [问题描述]，根因为 [简述根因]
```

> 版本号由用户指定，如用户未指定则使用 `[Unreleased]`。

### 步骤 8：生成任务交付文档

> ⚠️ **必须在所有步骤完成后执行**，使用 Write 工具将文档写入 `docs/delivery/` 目录。

文件命名规则：`docs/delivery/{YYYY-MM-DD}-bugfix-{问题简称}.md`（问题简称用英文小写短横线分隔，如 `null-pointer-on-login`、`order-amount-overflow`）。

文档模板如下（按实际情况填写，无变更的章节保留标题并标注"无"）：

```markdown
# 任务交付文档

## 基本信息
- **任务类型**: Bug 修复
- **执行日期**: YYYY-MM-DD
- **执行 Skill**: bug-fix
```
