# Go 需求澄清协议

> 本协议被以下 Go Skill 引用：`feature-dev-go`、`feature-dev-go-orchestrated`、`bug-fix-go`、`refactor-go`、`code-review-go`、`deploy-doc-go`、`testing-go`、`dev-review-go-orchestrated`
>
> 触发上述 Skill 时，必须先执行本协议，再进入正式阶段。
>
> 其中：
> - `feature-dev-go-orchestrated` 复用 `feature-dev-go` 的问题清单
> - `dev-review-go-orchestrated` 根据任务类型复用 `feature-dev-go` / `bug-fix-go` / `code-review-go` 的问题清单

## 核心原则

### 强制门控
1. 禁止在 Go 任务的关键输入未澄清前直接写代码、写测试或下结论。
2. 每轮只问最关键的 3~5 个问题，避免信息轰炸。
3. 已能从上下文合理推断的信息，可明确写出假设并请用户纠正。
4. 用户明确要求“先做再说”时，可带假设推进，但必须显式标注假设。

### 退出条件
- 当前 Skill 的全部必问项已明确或可合理推断。
- 剩余未知项不会显著影响实现质量、测试范围或交付结果。

## 执行流程

### 步骤 1：内部自检
收到请求后，先对照对应 Skill 的问题清单，在内部判断每项是：
- 已知
- 未知
- 可推断

### 步骤 2：结构化追问
把未知的必问项和高影响的选问项整理成一轮追问：

```markdown
在开始之前，我需要确认几个关键信息：

1. **[问题标题]**: [具体问题]
2. **[问题标题]**: [具体问题]
3. **[问题标题]**: [具体问题]

> 已理解的信息：[简要复述已知背景]
```

### 步骤 3：循环确认
- 用户回复后更新自检状态。
- 若仍有关键缺失，继续下一轮追问。
- 最多追问 3 轮；之后若仍有少量模糊点，可带假设推进。

### 步骤 4：确认需求理解
退出澄清时，用自己的话简要复述：
- 要做什么
- 影响哪些 Go 模块/包
- 是否有明确的技术约束

## 各 Skill 专属问题清单

### `feature-dev-go`

#### 必问项
| 编号 | 问题 | 说明 |
|:---:|------|------|
| FG-01 | 业务目标 | 功能解决什么问题，给谁使用 |
| FG-02 | 影响范围 | 涉及哪些包、模块、服务或接口 |
| FG-03 | 接口形态 | Gin HTTP、gRPC、定时任务、消息消费者或内部库 |
| FG-04 | 关键规则 | 校验、状态流转、幂等、并发、事务等核心规则 |

#### 选问项
| 编号 | 问题 | 说明 |
|:---:|------|------|
| FG-05 | 技术约束 | 是否必须使用 Gin / Fx / GORM / Viper 等既有栈 |
| FG-06 | 数据与存储 | 是否涉及表结构、索引、缓存、消息队列 |
| FG-07 | 性能要求 | 并发量、响应时间、超时、批量处理规模 |
| FG-08 | 安全要求 | 鉴权、脱敏、敏感配置、签名、金额精度 |
| FG-09 | 参考实现 | 是否有可复用的现有包或类似功能 |

### `bug-fix-go`

#### 必问项
| 编号 | 问题 | 说明 |
|:---:|------|------|
| BG-01 | 问题现象 | 报错、panic、错误返回、行为异常 |
| BG-02 | 复现条件 | 请求、数据、环境、并发条件、是否稳定复现 |
| BG-03 | 期望行为 | 正确结果应该是什么 |

#### 选问项
| 编号 | 问题 | 说明 |
|:---:|------|------|
| BG-04 | 影响范围 | 是否影响单接口、整服务或后台任务 |
| BG-05 | 最近变更 | 最近是否改过相关包、配置、依赖或中间件 |
| BG-06 | 运行信息 | 是否有日志、stack trace、pprof、监控指标 |
| BG-07 | 环境差异 | 本地、测试、生产是否都能复现 |

#### 选问项

### `refactor-go`

#### 必问项
| 编号 | 问题 | 说明 |
|:---:|------|------|
| RFG-01 | 重构目标 | 为什么要重构，是为了解耦、收敛职责、提取公共逻辑、便于测试还是降低复杂度 |
| RFG-02 | 重构范围 | 涉及哪些包、文件、结构体、接口或 Handler |
| RFG-03 | 行为约束 | 是否要求外部行为完全不变，是否允许顺带修 bug 或调整接口契约 |

#### 选问项
| 编号 | 问题 | 说明 |
|:---:|------|------|
| RFG-04 | 回归保护 | 当前有哪些测试、接口用例、benchmark 或人工验证手段可用于比对前后行为 |
| RFG-05 | 技术边界 | 哪些依赖注入、路由、配置结构、数据库访问路径或中间件接入方式不能改变 |
| RFG-06 | 并发约束 | 是否存在 goroutine、锁、channel、context 传播等不能被破坏的关键行为 |
| RFG-07 | 期望结果 | 希望重构后达到什么状态，如方法拆分、接口抽象、依赖倒置、重复逻辑收敛 |

## 追问话术建议

### 推荐
- “在开始之前，我需要确认几个和 Go 实现有关的关键信息：”
- “我目前理解这个需求会影响 `handler/service/repo` 这几层，如有偏差请纠正。”
- “关于技术栈我先按 Gin + Fx + GORM 理解，如果项目不是这样请告诉我。”

### 避免
- 一次性列十几个问题。
- 只问“还有什么补充吗”这种泛问题。
- 忽略 Go 特有约束，例如 `context.Context`、并发安全、错误处理链路。
