# AI 辅助开发工作流指南

> 本文档面向团队全体后端开发人员，说明如何在日常开发中利用 AI 编码助手提升效率与质量。
> **本框架与特定 AI 工具无关**，适用于 Cursor、GitHub Copilot、Gemini Code Assist、通义灵码等任意 AI 编码助手。

---

## 一、概述

### 1.1 什么是 AI 辅助开发框架

本框架通过三层机制，将团队的编码规范、安全标准和工作流程"内化"到 AI 助手中，使 AI 在辅助开发时自动遵守团队约定：

```
┌───────────────────────────────────────────────────────┐
│  层3: 本文档（给人看的说明书）                            │  ← 串联一切
├───────────────────────────────────────────────────────┤
│  层2: Skills — AI 的操作手册                            │  ← 定义"怎么干"
│    feature-dev / code-review / testing / deploy-doc    │
├───────────────────────────────────────────────────────┤
│  层1: Rules — AI 的铁律约束                             │  ← 定义"不能犯什么错"
│    00-global / 01-java-backend / 02-security / 03-api  │
└───────────────────────────────────────────────────────┘
```

### 1.2 三层各司其职

| 层级 | 用途 | 生效方式 | 类比 |
|------|------|----------|------|
| **Rules** | 约束底线，防止 AI 写出不符合规范的代码 | 自动或手动注入到 AI 上下文 | 公司规章制度 |
| **Skills** | 标准作业流程，指导 AI 按步骤完成特定任务 | 用户在对话中提出需求时触发 | 岗位操作手册 |
| **指南** | 告诉团队成员怎么使用这套体系 | 手动阅读 | 新人入职培训 |

### 1.3 框架文件结构

```
ai-dev-framework/
├── rules/                          ← 约束规范（Markdown 格式，通用）
│   ├── 00-global.md               # 全局强制规范
│   ├── 01-java-backend.md         # Java 后端架构与编码规范
│   ├── 02-security.md             # 安全红线
│   └── 03-api-design.md           # API 接口设计规范
│
├── skills/                         ← 标准作业流程（Markdown 格式，通用）
│   ├── feature-dev/SKILL.md       # 功能开发 SOP
│   ├── code-review/SKILL.md       # 代码评审 SOP
│   ├── testing/SKILL.md           # 测试生成 SOP
│   └── deploy-doc/SKILL.md        # 部署文档生成 SOP
│
└── docs/
    └── ai-dev-workflow.md          ← 本文档
```

## 二、适配不同 AI 工具

本框架的 Rules 和 Skills 本质上都是 **Markdown 文档**，内容与任何特定工具无关。各工具只是"加载方式"不同：

### 2.1 工具适配对照表

| AI 工具 | Rules 加载方式 | Skills 加载方式 | 备注 |
|---------|---------------|----------------|------|
| **Cursor** | 复制到 `.cursor/rules/`，扩展名改为 `.mdc` | 复制到 `.cursor/skills/` | 支持 globs 自动匹配 |
| **Gemini Code Assist** | 复制到 `.agents/skills/` 或写入 user_rules | 复制到 `.agents/skills/` | 支持 SKILL.md 格式 |
| **GitHub Copilot** | 写入 `.github/copilot-instructions.md` | 对话中手动引用 | 目前仅支持单文件指令 |
| **通义灵码** | 项目级提示词配置中粘贴 | 对话中手动引用 | 通过项目规范功能加载 |
| **其他工具** | 对话开始时将规则内容粘贴给 AI | 对话中将 SKILL.md 内容发送给 AI | 通用方式，所有工具适用 |

### 2.2 通用使用方式（适用于任何 AI 工具）

如果你的 AI 工具不支持自动加载规则，可以在每次对话开始时手动操作：

1. **加载规则**：将对应的规则文件内容（如 `01-java-backend.md`）粘贴到对话中，告诉 AI "请遵循以下编码规范"
2. **触发 Skill**：将对应的 `SKILL.md` 内容粘贴到对话中，告诉 AI "请按照以下流程执行"

## 三、Rules（规则文件）说明

### 3.1 文件清单

| 文件 | 建议激活条件 | 内容概要 |
|------|-------------|----------|
| `00-global.md` | **始终生效** | 语言要求、最小变更原则、格式保持 |
| `01-java-backend.md` | 编辑 Java 代码时 | 分层架构、命名规范、事务管理、日志规范 |
| `02-security.md` | 编辑代码或配置文件时 | 加密标准、认证授权、注入防护、暴露面收敛、依赖安全 |
| `03-api-design.md` | 编辑 Controller 层时 | 响应格式、入参校验、异常处理、HTTP 语义 |

### 3.2 如何扩展
- 新增规则：创建新的 `.md` 文件，遵循相同的格式
- 修改规则：直接编辑对应文件
- 命名建议：使用数字前缀排序（`04-xxx.md`、`05-xxx.md`）

## 四、Skills（技能）说明

### 4.1 技能清单

| 技能 | 触发关键词 | 用途 |
|------|------------|------|
| `feature-dev` | "开发XXX功能"、"实现XXX需求" | 从需求分析到编码实现的全流程 |
| `code-review` | "审查代码"、"review" | 安全/规范/质量三维度评审 |
| `testing` | "写测试"、"生成测试用例" | 系统性生成单元测试 |
| `deploy-doc` | "生成部署文档"、"需要什么部署操作" | 自动提取配置变更并生成部署清单 |

### 4.2 使用示例

#### 功能开发
```
用户: 帮我开发用户短信验证码登录功能

AI 执行 feature-dev Skill:
  → 阶段一: 输出实现方案（影响范围、接口设计、数据库变更）
  → 等待用户确认
  → 阶段二: 按 Entity → DTO → Repository → Service → Controller 顺序编码
  → 阶段三: 自检清单 + 变更摘要
```

#### 代码评审
```
用户: review 一下 UserService.java

AI 执行 code-review Skill:
  → 安全审查: 10 项检查
  → 规范审查: 8 项检查
  → 质量审查: 10 项检查
  → 输出评审报告表格
```

#### 测试生成
```
用户: 给 UserServiceImpl 写单元测试

AI 执行 testing Skill:
  → 分析目标代码的方法和分支
  → 输出测试用例设计表
  → 等待用户确认
  → 按 Given-When-Then 结构编写 JUnit 5 + Mockito 测试
```

#### 部署文档生成
```
用户: 生成这次改动的部署文档

AI 执行 deploy-doc Skill:
  → 扫描 yml/pom.xml/SQL 等文件变更
  → 提取环境变量、配置项、DDL、依赖变更
  → 生成标准格式的部署清单
```

## 五、标准开发流程

以下是结合 AI 辅助的**推荐日常开发流程**：

```
  ┌──────────┐
  │ 1. 接到需求 │
  └─────┬────┘
        ↓
  ┌──────────────────────────┐
  │ 2. 告诉 AI "开发XXX功能"    │  ← 触发 feature-dev Skill
  │    AI 输出实现方案          │
  └─────┬────────────────────┘
        ↓
  ┌──────────────────────────┐
  │ 3. 人工审核方案             │  ← 确认方向正确
  │    确认后 AI 开始编码       │     Rules 自动约束代码质量
  └─────┬────────────────────┘
        ↓
  ┌──────────────────────────┐
  │ 4. 编码完成                │
  │    告诉 AI "review 代码"   │  ← 触发 code-review Skill
  └─────┬────────────────────┘
        ↓
  ┌──────────────────────────┐
  │ 5. 修复评审发现的问题       │
  └─────┬────────────────────┘
        ↓
  ┌──────────────────────────┐
  │ 6. 告诉 AI "写测试"        │  ← 触发 testing Skill
  │    AI 生成单元测试          │
  └─────┬────────────────────┘
        ↓
  ┌──────────────────────────┐
  │ 7. 告诉 AI "生成部署文档"   │  ← 触发 deploy-doc Skill
  │    AI 生成部署清单          │
  └─────┬────────────────────┘
        ↓
  ┌──────────────────────────┐
  │ 8. 提交 MR                │
  │    附带: 评审报告 + 部署文档 │
  └──────────────────────────┘
```

## 六、注意事项

### 6.1 AI 不是银弹
- AI 生成的代码**必须人工审核**，不能盲目信任
- AI 的方案设计**需要人工确认**后才能进入编码
- 安全敏感的代码（加密、认证、资金操作）**必须有资深开发二次审核**

### 6.2 规则的维护
- 规则文件由**技术负责人**统一维护，避免随意修改
- 重大规则变更需要团队评审
- 建议每季度 review 一次规则，移除过时条目、补充新的安全要求

### 6.3 版本管理
- 规则和 Skill 文件应当**纳入 Git 版本控制**，确保团队统一
- 规则文件的修改应当通过正常的 MR 流程审批，**禁止未经评审直接推送变更**
- 建议通过 CODEOWNERS 限制规则文件的修改权限

### 6.4 不适合 AI 的场景
- 涉及核心架构决策的工作，需要人类架构师主导
- 涉及敏感业务规则的，AI 只能辅助实现，逻辑必须人工确认
- 紧急线上问题排查，AI 可辅助分析日志，但决策权在人

## 七、常见问题 FAQ

**Q: 规则内容与特定 AI 工具绑定吗？**
A: 不绑定。规则和 Skill 本质是 Markdown 文档，任何 AI 工具都可以读取和遵守。只是不同工具的"加载方式"不同，参见第二章适配表。

**Q: 可以同时使用多个 Skill 吗？**
A: 建议一次专注触发一个 Skill。完成后再触发下一个。

**Q: 规则之间冲突怎么办？**
A: 安全规则（`02-security.md`）优先级最高，如有冲突以安全规则为准。

**Q: 如何添加前端（Vue/React）的规则？**
A: 新增 `04-frontend.md`，按团队前端技术栈编写规范内容。

**Q: 团队用的不是 Cursor 怎么办？**
A: 参见第二章适配表。最通用的方式是在对话开始时将规则内容粘贴给 AI。
