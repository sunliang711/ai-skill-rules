# Bug 修复 SOP（Standard Operating Procedure）

> 当用户提出"修复 Bug"、"排查问题"、"某某功能报错"、"线上异常"等 Bug 修复类请求时，严格按以下流程执行。

---

## 阶段一：问题收集与复现

### 步骤 1：信息收集
收集以下关键信息，**信息不充分时必须主动询问**：

| 信息项 | 说明 |
|--------|------|
| 现象描述 | 具体表现是什么（报错信息、异常行为、与预期的差异） |
| 复现步骤 | 操作路径、触发条件、是否必现 |
| 影响范围 | 影响哪些用户/功能/环境（生产/测试/开发） |
| 错误日志 | 关键堆栈信息、错误码、日志片段 |
| 变更历史 | 问题首次出现的时间，是否与某次发布关联 |
| 已尝试方案 | 用户或团队已经排查过的方向和结论 |

### 步骤 2：问题复现与定位
1. 根据用户描述定位相关代码文件和模块
2. 阅读并理解相关代码的业务逻辑
3. 如有错误日志，从堆栈信息反向追踪调用链
4. 如无明确日志，根据业务流程正向梳理数据流

---

## 阶段二：根因分析

### 步骤 3：系统性排查
按以下类别逐项排查，**必须找到根因后才能进入修复阶段**：

#### 3.1 代码逻辑类
- 条件判断错误（边界值、逻辑运算符）
- 空指针 / 空集合未处理
- 循环逻辑错误（死循环、off-by-one）
- 类型转换异常
- 状态流转遗漏

#### 3.2 数据类
- 数据库字段类型/长度不匹配
- 数据精度丢失（浮点运算、时区转换）
- 脏数据 / 不一致数据
- SQL 逻辑错误（JOIN 条件、WHERE 过滤）

#### 3.3 并发与资源类
- 竞态条件 / 线程安全问题
- 死锁 / 超时
- 连接池耗尽 / 资源泄漏
- 缓存不一致

#### 3.4 集成与配置类
- 接口契约变更（字段名、类型、必填项）
- 环境配置差异（开发 vs 生产）
- 依赖版本不兼容
- 网络/中间件连接问题

### 步骤 4：输出分析结论（必须输出）
确认根因后，输出以下分析报告，**等待用户确认后再进入修复阶段**：

```markdown
## Bug 分析报告

### 1. 问题概述
- 现象: [一句话描述]
- 严重等级: [🔴 阻断 / 🟡 较高 / 🟢 一般]
- 影响范围: [受影响的功能/用户/环境]

### 2. 根因定位
- 问题分类: [代码逻辑 / 数据 / 并发与资源 / 集成与配置]
- 根因文件: [文件路径:行号]
- 原因说明: [详细解释为什么会出现这个问题]

### 3. 修复方案
- 修复策略: [描述具体的修复思路]
- 涉及文件: [列出需要修改的文件清单]
- 潜在风险: [修复是否可能引入新问题，需要注意什么]
```

---

## 阶段三：修复实现

### 步骤 5：编码修复
1. **只修改与 Bug 直接相关的代码**，严禁"顺手"重构
2. 修复代码必须沿用原文件的编码风格
3. 如果修复涉及多个文件，按依赖关系从底层到上层依次修改
4. 在修复点添加简要注释说明修复原因（仅限非显而易见的修复）

### 步骤 6：防御性加固（视情况）
在不改变原有逻辑的前提下，可建议以下加固措施：
- 添加空值检查
- 补充参数校验
- 增加异常捕获和友好提示
- 添加关键日志便于后续排查

**注意**：加固措施仅作为建议提出，用户确认后方可添加。

---

## 阶段四：验证与交付

### 步骤 7：自检清单
修复完成后，逐条对照以下清单自检：

#### 修复正确性
- [ ] 根因已消除，非临时绕过
- [ ] 修复逻辑与分析报告中的方案一致
- [ ] 边界情况已覆盖（空值、极值、异常输入）

#### 影响评估
- [ ] 修改未破坏原有功能的正常流程
- [ ] 修改未影响其他模块的调用方
- [ ] 数据库变更（如有）兼容已有数据

#### 代码质量
- [ ] 无新增 Lint 错误或编译警告
- [ ] 无遗留调试代码（System.out、console.log、硬编码测试数据）
- [ ] 修复代码风格与原文件保持一致

### 步骤 8：输出变更摘要

```markdown
## 修复摘要

### Bug 描述
[一句话描述原始问题]

### 根因
[一句话描述根本原因]

### 修复内容
- 修改文件: [列出修改的文件路径和主要改动]
- 修改行数: 约 N 行

### 验证建议
- [列出验证此修复的具体操作步骤]
- [如有必要，列出需要回归测试的相关功能]
```

## 特殊情况处理

### 无法确定根因
如果排查后仍无法确定根因：
1. 输出已排查的方向和排除的可能性
2. 提出进一步排查所需的信息（更详细的日志、数据样例、环境信息）
3. **严禁在根因不明的情况下盲目修改代码**

### 临时规避方案
如果根因修复周期较长且问题紧急：
1. 可提出临时规避方案（workaround），但必须明确标注为临时方案
2. 同时输出根本修复方案的思路和预估工作量
3. 在代码中添加 `// TODO: [日期] 临时方案，待根本修复后移除` 注释
