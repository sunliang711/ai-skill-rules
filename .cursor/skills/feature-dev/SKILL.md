# 功能开发 SOP（Standard Operating Procedure）

> 当用户提出"开发XXX功能"、"实现XXX需求"、"新增XXX接口"等功能开发类请求时，严格按以下流程执行。

---

## 阶段一：需求理解与方案设计

### 步骤 1：需求确认
1. 仔细阅读用户提供的需求描述
2. 如有不明确的地方，**必须主动询问**，特别是：
   - 业务边界（哪些情况需要处理、哪些不需要）
   - 权限要求（哪些角色可以操作）
   - 性能要求（并发量、响应时间）
   - 数据要求（字段精度、长度限制）
3. **禁止在需求不清晰时自行假设并开始编码**

### 步骤 2：方案设计（必须输出）
确认需求后，先输出以下设计文档，**等待用户确认后再进入编码阶段**：

```markdown
## 实现方案

### 1. 影响范围
- 涉及模块: [列出涉及的业务模块]
- 新增文件: [列出需要创建的文件清单]
- 修改文件: [列出需要修改的现有文件]

### 2. 数据库变更（如有）
- 新增表: [表名、核心字段]
- 修改表: [表名、变更字段]
- 新增索引: [索引说明]

### 3. 接口设计
| 路径 | 方法 | 功能 | 入参 | 出参 |
|------|------|------|------|------|

### 4. 核心业务逻辑
[用文字描述关键的处理流程、校验规则、状态流转等]

### 5. 新增依赖（如有）
| GroupId | ArtifactId | 版本 | 用途 |
|---------|------------|------|------|

### 6. 配置项变更（如有）
- 新增环境变量: [变量名 / 用途 / 是否敏感]
- 新增配置项: [配置路径 / 默认值 / 说明]

### 7. 安全考量
- 权限要求: [需要什么角色/权限]
- 输入校验: [需要哪些特殊校验]
- 脱敏要求: [日志和响应中需要脱敏的字段]
```

### 步骤 3：用户确认
- **等待用户确认方案无误后，方可进入编码阶段**
- 如果用户提出修改意见，更新方案后再次确认

---

## 阶段二：编码实现

### 步骤 4：按分层顺序编码
严格按照以下顺序逐层实现，**禁止跳跃**：

#### 4.1 数据库脚本（如有变更）
- 编写 DDL 脚本（建表、加字段、加索引）
- 脚本放在约定的 `sql/` 或 `db/migration/` 目录下

#### 4.2 Entity / DO 层
- 创建数据库映射实体类
- 字段类型与数据库保持一致
- 添加表名、字段的注释说明

#### 4.3 DTO 层
- 创建请求 DTO（`XxxReqDTO`）和响应 DTO（`XxxRespDTO`）
- 请求 DTO 必须添加 `@NotNull`、`@NotBlank`、`@Size` 等校验注解
- 响应 DTO 中敏感字段必须脱敏注解或排除

#### 4.4 Repository / Mapper 层
- 创建数据访问接口
- SQL 全部使用参数化查询（MyBatis `#{}` / JPA Named Parameter）
- 分页查询必须有 `MAX_PAGE_SIZE` 限制

#### 4.5 Service 层
- 先定义接口（`XxxService`），再实现（`XxxServiceImpl`）
- 业务逻辑全部在 Service 层完成
- 事务边界通过 `@Transactional` 控制
- 异常使用自定义 `BusinessException` 抛出

#### 4.6 Controller 层
- 添加权限注解（默认需要认证，公开接口显式标注 `@PermitAll`）
- 入参标注 `@Valid`
- 仅做参数接收和响应封装，**禁止包含业务逻辑**
- 添加 Swagger 注解（`@Operation`、`@Schema`）

### 步骤 5：代码注释
- 类头部添加功能说明注释
- 核心业务逻辑添加行内注释说明"为什么这样做"
- 复杂算法或规则添加块注释说明

---

## 阶段三：自检与交付

### 步骤 6：自检清单
编码完成后，必须逐条对照以下清单自检，并在回复中明确标注每项的通过状态：

#### 架构规范
- [ ] Controller 层无业务逻辑
- [ ] Service 层无直接数据访问（通过 Repository）
- [ ] DTO 与 Entity 严格分离
- [ ] 无跨层调用

#### 命名规范
- [ ] 类名、方法名、变量名符合团队命名规范
- [ ] 无魔法数字（硬编码数值已提取为常量）
- [ ] REST 路径格式正确（小写、连字符、复数）

#### 安全规范
- [ ] 接口有权限注解
- [ ] 入参有 `@Valid` 校验
- [ ] 日志无敏感信息
- [ ] 无硬编码凭据
- [ ] 异常处理已脱敏

#### 健壮性
- [ ] 空值已处理
- [ ] 集合操作有边界检查
- [ ] 分页查询有上限
- [ ] 资源（连接/流）正确关闭

### 步骤 7：变更摘要
自检通过后，输出简洁的变更摘要：

```markdown
## 变更摘要
- 新增文件: N 个
- 修改文件: N 个
- 新增接口: [列出路径和功能]
- 数据库变更: [有/无，简述]
- 新增配置项: [有/无，列出]
- 新增环境变量: [有/无，列出]
```
